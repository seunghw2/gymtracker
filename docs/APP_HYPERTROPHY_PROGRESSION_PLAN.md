# 앱 근비대 점진적 과부하 적용 방안 (GymTracker)

> **한 줄 요약** — 이 앱은 "양(주간 부위 볼륨)·질(같은 무게 반복/RIR 개선)" 두 축으로 진행을 판정하고, "직전 기록 + 작은 bump"로 처방하며, 최고·PR은 동기·맥락용으로만 둔다. 처방 숫자는 결정론 엔진이 단일 소스로 내고 AI는 그 "왜"만 통역한다. 화면·데이터는 대부분 이미 깔려 있고, 남은 일은 **볼륨을 1차로 끌어올리고 1RM을 강등하는 비중 재배치 + 탑세트/백오프 분리 판정 + RIR 양방향 보정**이다.

**작성일**: 2026-06-29

## 사용자 확정 방향 (모든 제안의 전제)

1. **진행 기준 = 직전 기록 + 주간 볼륨 중심**(근비대). 최고 기록은 맥락·동기용.
2. **세트 구성 = 종목별 진행 룰 선택**(더블 프로그레션 / 탑세트+백오프 / 총 반복). 사용자는 **탑세트+백오프** 선호.
3. **RIR 수집 = 선택형**(기본 OFF, 원할 때만).
4. **AI = 보조**. 결정론 엔진이 처방, AI는 설명·동기·질문답변·메모 요약(환각 최소).

**대상 가정**: 자연인 · 중급 · 근비대 목표 · 상업 헬스장(머신이 매번 다를 수 있음).

**근거 문서**: `HYPERTROPHY_PROGRESS_AND_TRACKING.md`(가장 중요), `PROGRESSIVE_OVERLOAD_GUIDE.md`, `PROGRESSION_TRAINING_AND_LOGGING.md`, `TRAINING_PROGRAMS_ENCYCLOPEDIA.md`, `PROGRESSIVE_OVERLOAD_RESEARCH.md`, `APP_VS_GUIDE_AUDIT.md`, `FINAL_PRODUCT_SPEC.md`, `APP_AUDIT.md`, `PROGRESSION_ENGINE_SPEC.md`. 실제 코드: `ProgressionEngine.java`, `ExerciseGoal.java`, `GoalSetting.java`, `OverloadService.java`, `SetType.java`, `Effort.java`, `app/(tabs)/index.tsx`, `app/workout.tsx`, `app/exercise/[name].tsx`.

> **코드 현황 정정(중요)**: 일부 감사 문서는 "RIR을 수집만 하고 폐기한다"고 적었으나, 현재 코드는 노력도(`Effort`)를 수집·저장하고 `readyToIncrease`의 `rpe` 트리거에 이미 연결돼 있다(`ProgressionEngine.java:106-120`). 반대로 `SetType`에는 아직 `TOP_SET`/`BACKOFF` 태그가 없다(`NORMAL/WARMUP/DROP/FAILURE`만). 본 문서는 코드를 1차 근거로 삼는다.

# Q1 — 이 앱에서 근비대 점진적 과부하를 어떻게 활용할 것인가 (엔진 큰 그림)

## 1-1. 한 줄 설계

> **엔진은 "양·질 두 축"으로 판정하고, "직전 기록 + 작은 bump"로 처방한다.** 양 = 부위 주간 유효 볼륨(hard 세트 수), 질 = 같은 무게에서 반복/RIR 개선. 1RM은 처방 기준이 아니라 동기·맥락용 화폐로만 둔다.

`HYPERTROPHY_PROGRESS_AND_TRACKING.md` 1부의 판정 우선순위(① 주간 hard 세트 ② 실패 근접도 ③ 같은 무게 반복/RIR ④ 톤수·e1RM은 대리지표)를 앱의 **두 레이어**로 가른다.

| 레이어 | 담당 질문 | 데이터 단위 | 현 구현 | 비고 |
|---|---|---|---|---|
| **종목 레이어 (질·강도)** | "이 종목이 직전보다 나아졌나?" | DayRecord(한 날 한 종목 세트) | `ProgressionEngine.compute`/`evaluate` (구현됨) | 직전 비교 + 작은 bump |
| **부위 레이어 (양)** | "이 부위가 주간 충분히 자극됐나?" | 주간 부위 hard 세트 수 | `weeklySetsByMuscle` (구현됐으나 워밍업 미필터·RIR 게이트 없음 → `APP_VS_GUIDE_AUDIT` 심각2) | MEV 4~6 / 성장 10~20 |

핵심: **현 엔진은 "종목 레이어"만 1차로 본다.** 근비대 진짜 동인인 "부위 주간 볼륨"은 보조로 밀려 있고 집계도 워밍업을 포함해 과대다. Q1의 목표는 **이 두 축을 종목 카드와 별개로 동시에 노출**하는 것.

## 1-2. 양·질 매트릭스를 앱 상태 라벨로

| | 같은 무게 반복/RIR 개선 O | 개선 X |
|---|---|---|
| **주간 부위 볼륨 ↑/유지** | ✅ `GROWING` (확실한 성장) | ⚠️ `VOLUME_BUBBLE` (볼륨 거품 의심 — RIR 점검 유도) |
| **주간 부위 볼륨 ↓** | 🟡 `STRENGTH_ONLY` (강도↑·볼륨 회복 필요) | ❌ `STALL_REVIEW` (정체 — 현 엔진 분기) |

```
판정(종목, 주간부위):
  질개선 = evaluate(prev,cur).result == IMPROVED
           OR (같은무게 같은반복인데 RIR이 EASY 쪽으로 이동)
  양추세 = 부위주간 hard세트 4주 이동평균 방향   // 종목 아닌 '부위' 단위

  if 양추세 in {UP, FLAT} and 질개선:      → GROWING
  if 양추세 in {UP, FLAT} and not 질개선:  → VOLUME_BUBBLE   // "세트는 늘었는데 같은 무게가 안 쉬워졌어요. RIR 확인"
  if 양추세 DOWN and 질개선:               → STRENGTH_ONLY   // "강해졌어요. 볼륨은 한 세트 더 채울 여지"
  if 양추세 DOWN and not 질개선:           → STALL_REVIEW (기존 stalling 게이트 적용)
```

이 매트릭스는 **톤수만 보고 "성장"이라 속는 함정**(`HYPERTROPHY_…` 사례 A·B)을 구조로 막는다. `VOLUME_BUBBLE`이 "75kg 8회가 RIR 4~5로 쉬웠던" 거품 케이스를 잡는 상태다.

## 1-3. 엔진이 매 순간 무엇을 기준 삼고 무엇을 처방하나

| 시점 | 기준(읽는 것) | 처방(주는 것) |
|---|---|---|
| **오늘 목표** | `getLastComparableRecord` (직전 1건) | 직전 총 + bump(맨몸+1 / 무게+1~2). **최고·PR 절대 안 씀** |
| **현재 위치** | `getBestComparableRecord` (최고 비교) | "이전 최고 근처/돌파" 맥락 표시(처방 아님) |
| **동기** | `getAllTimePR` (역대 최고 단일세트) | "역대 PR" 배지. 근비대 정체 판정에 단독 사용 금지 |
| **주간 양** | 부위 hard 세트 4주 추세 | 부족 부위 "+1~2세트" 제안 |

→ **세 기록의 역할을 절대 섞지 않는다**(`PROGRESSION_ENGINE_SPEC` §0-1). 현 코드는 이 3분리를 DTO로 이미 잘 지킨다. Q1이 추가할 것은 **4번째 줄(부위 양)을 종목 카드 위 부위 대시보드로 병치**하는 것.

# Q2 — "최고 기록" 판정 + 증량(반복/중량) 제안 로직

## 2-1. "최고 기록"은 종목 신뢰 등급에 따라 의미가 다르다

`HYPERTROPHY_…` 5부 **3계층 추적**을 데이터와 결합한다. 현 `ExerciseGoal.role`(core/support/log_only)은 "추적 강도"만 표현하고 **"무게 신뢰도"는 표현하지 못한다.** 핵심 제안: `role`에 **신뢰 계층(anchorTier)을 직교 축으로 추가**.

| anchorTier | 정의 | 예 | "최고 기록"의 의미 | e1RM 신뢰 | 진행 판정 |
|---|---|---|---|---|---|
| **ANCHOR** | 프리웨이트 복합·체중 (어디서나 동일) | 백스쿼트·벤치·데드/RDL·OHP·바벨로우·풀업 | **무게×반복 정밀 비교 가능. PR 신뢰** | 저반복서 높음 | 무게로 정밀 |
| **MAIN_MACHINE** | 자주 가는 헬스장 머신, 세팅 고정 | 그 헬스장 랫풀·레그프레스 | **세팅 동일할 때만 비교** | 보조 | 무게+반복(세팅 메모 전제) |
| **VARIABLE** | 가변 머신·케이블·출장 | 처음 보는 시티드로우 | **무게 PR 무의미. 비교 포기** | 의미없음 | 볼륨+RIR만 |

```
anchorTier 기본값:
  ruleType == barbell_main  → ANCHOR
  ruleType == bodyweight    → ANCHOR        // 체중은 어디서나 동일
  ruleType == machine_cable → MAIN_MACHINE  // 사용자가 VARIABLE로 다운그레이드 가능
  ruleType == isolation     → MAIN_MACHINE  // 덤벨 고립이면 ANCHOR로 업그레이드 권장
```

**판정 결과(Q2 핵심):**
- **ANCHOR** → `getAllTimePR`·`getBestComparableRecord` 그대로 신뢰. "역대 최고 82.5kg×6" 배지 유효.
- **VARIABLE** → "최고 기록(무게)"을 **종목 카드에서 숨기거나 회색 처리**, 대신 "이 부위 주간 볼륨 기여"로 대체. `HYPERTROPHY_…` 5부의 질문 전환(이 종목이 늘었나? → 이 부위가 늘었나?)을 UI로 강제.

> 현 갭: `allTimePrOf`는 ruleType 무관하게 무게 PR을 계산한다. 가변 머신에서도 "최고 82kg"가 떠 다른 머신 비교를 유도하는 오도가 생긴다. anchorTier=VARIABLE이면 PR 화폐를 끄는 게 Q2의 명시적 처방.

## 2-2. e1RM·톤수·실측의 한계 (무엇을 믿고 무엇을 안 믿나)

| 지표 | 믿는 조건 | 못 믿는 경우 | 앱에서의 역할 |
|---|---|---|---|
| **e1RM(Epley)** | ANCHOR + 저반복(≤6~8) | 머신(레버리지 변동 수십%) / 고반복(추정 오차 큼) | ANCHOR 강도 추세 보조선. **헤드라인 금지**(현 종목상세 헤드라인이 1RM — 강등 대상) |
| **톤수(Σ무게×반복)** | RIR 일정할 때 | RIR 헐거워지면 거품(사례 B) | 추세 참고. 단독 판정 금지 → `VOLUME_BUBBLE` 경고 |
| **실측 머신 무게** | 같은 머신·세팅 | 브랜드·도르래·핀단위 다르면 무의미 | VARIABLE은 "참고용·머신 다름" 라벨 |

## 2-3. 증량 제안 로직 — 무게 vs 반복 중 무엇을 올리라 할지

**항상 직전 대비 작은 증가**, **반복이 먼저, 무게는 게이트 통과 후**.

```
제안(종목, 직전기록 prev, 룰):
  [range_min, range_max] = 룰의 반복 범위
  sets = 룰의 목표 세트수
  w    = topWeight(prev)

  // 1. 증량 게이트: 모든 본세트가 범위 상단 도달했나? (더블 프로그레션)
  if metTop(prev, range_max, sets) AND 트리거충족(single/two_sessions/rpe):
     → INCREASE_LOAD
       newW = w + effectiveIncrement(룰)           // 머신/고립은 절반 마이크로로딩
       제안 = "{newW}kg · 반복은 하단 {range_min}회부터 재시작"   // ← 반복 리셋 명시
  else:
     → BUILD_REPS
       제안 = "{w}kg · 총 {progressionReps(prev)+1}~{+2}회"      // 무게 고정, 반복만 +1~2

  // 2. 안전 가드
  if 사용자입력 newW > w * 1.1:               경고 "도약 — 5% 이내 권장"
  if 무게↑인데 총반복 직전대비 30%↓:          경고 "폼 붕괴 가능 — 무게 되돌리기"
```

**증량폭(effectiveIncrement) — 현 구현 유지:**

| 종목 | 증량 단위 | 근거 |
|---|---|---|
| 상체 바벨 | +1.25~2.5kg (incUpper) | 작은 점프 |
| 하체 바벨 | +2.5~5kg (incLower) | 큰 근육 |
| 머신/케이블·고립 | base의 절반, 최소 0.5kg (`effectiveIncrement`) | 핀 큰 점프 방지 마이크로로딩 |

**"무게 vs 반복" 한 줄 규칙**: 평소엔 **반복**(같은 무게 +1~2회 누적), 모든 본세트가 상단을 채우면 **그때 무게** 한 칸 + 반복 하단 리셋. `READY_TO_INCREASE`에서 "소폭 증량 가능"만 주는 현 구현은 **새 무게·반복 리셋 수치를 직접 산출하지 않아 행동 불가**(`APP_VS_GUIDE_AUDIT` 심각4) → Q2가 보강할 지점.

## 2-4. 머신 변동 대응 — RIR 자가조절로 무게 신뢰도 보완

VARIABLE 종목은 무게 PR을 버리고 **"같은 RIR에서 반복/무게가 늘면 진행"**으로만 본다. RIR이 선택형이므로:
- RIR 입력됨 → "직전과 같은 RIR(2)에서 반복↑" 판정 가능.
- RIR 미입력 → 그 종목은 **"부위 주간 볼륨 1세트 적립"만** 확인하고 진행 판정은 ANCHOR에 위임(단정 보류).

# Q3 — 여러 훈련 방법을 앱에서 어떻게 활용·선택하게 하나

## 3-1. 핵심 결론 (먼저)

가이드(GUIDE 3-2~3-4)와 백과(ENCYCLOPEDIA)가 말하는 6개 방법은 **6개 엔진이 아니라, 하나의 이중 진행 엔진을 3파라미터로 분기한 것 + 세트구조 2종 + 직교 레버**다.

| 구분 | 방법 | 본질 |
|---|---|---|
| **공통 1차 엔진** | 이중 진행(더블 프로그레션) | 반복 채우면 무게↑·반복 리셋. 4유형 전부의 토대 |
| 엔진 변형(다이얼) | 반복수 우선 / 무게 우선 / 총 반복 진행 | 같은 엔진의 범위 폭·증량 게이트 차이 |
| 직교 레버 | 세트 추가(볼륨 진행) | 정체·고립/머신 1순위 돌파, 주간 볼륨 직접↑ |
| 직교 조절 | RIR 자기조절 | 증량 트리거를 "반복 채움"에서 "RIR 여유"로 |
| **세트 구조** | 탑세트+백오프 | 한 종목에 강도축·볼륨축 분리 (→ Q6) |

따라서 **사용자에게 6개를 펼쳐 고르게 하지 않는다.** 종목 유형(ruleType)으로 기본값을 자동 선택하고, 한 종목에 "진행 룰" 한 줄만 노출한다(헬스장 3초 룰).

## 3-2. 종목 유형별 기본 방법 매핑 (자동 선택)

**굵게 = 현 코드(`ProgressionEngine`)에 이미 구현된 기본값.**

| ruleType (Kind) | 기본 방법 | 반복 범위(비대) | 증량 게이트 | 증량폭 | 정체 1순위 출구 |
|---|---|---|---|---|---|
| **BARBELL_COMPOUND** (바벨=무게) | 이중진행 · **무게 우선** | **6~12** | 전 세트 상단 도달 | 상체 +2.5 / 하체 +2.5~5kg | 세트 추가 → 재정체 시 5~10%↓ |
| **MACHINE_OR_CABLE** (머신=반복) | 이중진행 · **반복수 우선** | **8~12** | 반복 상단 채움 후 핀 1단 | **마이크로(increment 절반, 최소 0.5kg)** | **세트 추가 우선** |
| **ISOLATION** (고립=마이크로/세트) | 이중진행 · **반복수 우선 + 마이크로로딩** | **8~15** | 상단 + 마지막 세트 여유 | **+0.5~1.25kg 또는 세트 추가** | 세트 추가 우선 |
| **BODYWEIGHT** (맨몸=총반복) | **총 반복수 진행** | 5~12(풀업/딥 1~5) | 임계 반복 도달 | 반복 +1~2 → 임계 후 +2.5~5kg 가중 | 같은 목표 재시도 |

판정 기준은 확정 방향대로 **직전 기록 + 주간 볼륨 중심**: 오늘 목표 = `직전 총 + 작은 bump`(맨몸 +1, 무게 +1~2), 최고/PR은 맥락·동기용.

## 3-3. "진행 룰" 선택 UI — 6개를 2층으로 접기

종목 카드에 펼치는 건 **2개 결정**뿐. 나머지는 ruleType이 자동으로 정한다.

```
[종목 설정] 바벨 백스쿼트  · 유형: 바벨 복합 (자동)
─────────────────────────────────
진행 방식        ▸ 탑세트 + 백오프        ▾   ← 결정 1 (세트 구조)
  · 이중 진행(기본)
  · 총 반복 진행
  · 탑세트 + 백오프      ← 사용자 선호, 바벨/메인에 기본 추천
증량 트리거      ▸ 2세션 연속 충족        ▾   ← 결정 2 (RIR 선택형)
  · 반복 채우면(기본)
  · 2세션 연속(2-for-2)
  · RIR 여유(선택 — 메인만 권장)
```

- **결정 1(세트 구조)**: ruleType에 따라 기본 추천이 다름 — 바벨/메인 컴파운드는 탑세트+백오프 기본 추천(사용자 선호·Madcow/FSL 패턴), 머신/고립은 더블 프로그레션 또는 총 반복 진행. "세트 추가"는 메뉴에 안 넣고 **정체 시 엔진이 제안하는 레버**로 둔다.
- **결정 2(증량 트리거)**: 현 코드에 `single/two_sessions/rpe`가 이미 연결됨(기본 two_sessions). RIR은 **선택형**(기본 OFF). 초보 함정(RIR 4~5 과소평가)을 막기 위해 RIR은 메인 컴파운드 위주 권장.

## 3-4. Q3에서 손봐야 할 현 구현 빈칸 (audit 근거)

1. **세트 추가가 처방에 미반영**(sets=3 하드코딩, `compute` 부근). 정체 1순위 레버인데 텍스트로만 떠 비활성 → `targetSets`를 실제 +1 산출해야 "볼륨 진행" 방법이 살아난다.
2. **이중진행 게이트 ↔ 매일 목표 변수 불일치**: 게이트는 "전 세트 상단", 매일 목표는 "총합 +N" → "총 반복 채웠는데 왜 증량 안 떠?" 괴리. 매일 목표를 세트별 상단 진척("12/12/10 → 마지막 12")으로 통일 권장.

# Q4 — 홈 화면 / 운동 중 화면

> **한 줄 결론**: 화면 골격은 이미 확정 방향에 가깝다. 문제는 "무엇을 1차로 띄우느냐"의 비중 배분이다. 세 화면 모두 지금은 **무게/1RM이 진행의 얼굴**이고 볼륨·직전 비교·RIR 개선이 보조다. 이를 뒤집는 **배치·라벨·강조의 재배치**가 핵심이며, 신규 데이터는 거의 RIR 저장 하나뿐이다(나머지는 이미 DTO에 존재).

## A. 홈 화면

### A-1. 직무 (3초 룰)
홈은 두 질문에만 답한다: ① **오늘 뭐부터 손대나** ② **이번 주 부위별로 잘 채우고 있나(10~20세트)**. 최고 기록·차트·동기 문구는 종목 상세·리포트로 미룬다.

### A-2. 현재 홈 vs 제안

| 요소 | 현재 | 제안 | 더(＋)/덜(－) |
|---|---|---|---|
| 블록 순서 | ① 확인 필요 → ② 이번 주 → ③ 종목 | **① 이번 주 부위 볼륨 → ② 오늘 확인 필요 → ③ 종목** | 부위 게이지를 최상단으로 |
| 볼륨 세트 정의 | 워밍업·드롭 포함 COUNT(심각2) | **워밍업/드롭 제외 = 힘든 세트만 카운트** | 과대집계 제거 |
| 부위 볼륨바 | 있음(충족 시 초록) | **유지 + MEV 4~6 / 목표 10~20 구간 마커** | ＋ 구간 의미 부여 |
| 오늘 확인 필요 | 증량준비+정체를 중립 묶음 | 유지. **각 행에 직전 기록 1줄 병기** | ＋ "이겨야 할 숫자" |
| 종목 카드 | 단계배지+지난기록+오늘목표+주의 | 유지. **1RM/최고 기록 헤드라인 금지** | － 동기 숫자는 상세로 |

### A-3. 블록1 — 이번 주 부위별 볼륨 (최상단, 게이지)

머신이 매번 바뀌어도 "세트 수"는 안 변하므로 **머신 변동에 면역인 유일한 진행 신호**다.

```
이번 주
가슴   ▓▓▓▓▓▓▓▓░░░░  12/16 세트
등     ▓▓▓▓▓▓░░░░░░   9/14 세트   ← 부족
하체   ▓▓▓▓▓▓▓▓▓▓▓░  15/16 세트
어깨   ▓▓▓▓░░░░░░░░   6/12 세트   ← 부족
부족한 부위 · 등 5세트, 어깨 6세트     출석 ●●●○○  3/5회
```

- 게이지 구간: 0~MEV(~5)는 회색, MEV~10은 노랑, 10~20 성장 구간은 초록. "그냥 채움"이 아니라 "성장 구간 안에 있나"를 색으로.
- 부족 부위는 **남은 세트 수**로 표기 → 바로 행동. 데이터: `bodyPartVolumes`(이미 존재) + 카운트 필터 교정.

## B. 운동 중 화면

### B-1. 직무 (눈 0.5초)
종목 카드 위 **1줄 배너 하나**: "오늘 목표 + 남은 반복". 단계 라벨·성공조건은 배너 탭(시트)으로 미룬다. 현 코드 이미 구현.

### B-2. 세트행 와이어 (탑세트+백오프 구분)

```
벤치프레스
┌─ 오늘 82.5kg 총 24회 · 남은 8회 ───  (탑세트 기준) ──┐  ← 배너(탭 시 시트)
SET  무게(kg)   횟수   ✓
 1   [82.5] 탑   8     ✓   지난 80×8
 2   [82.5] 탑   8     ✓   지난 80×8
 3   [82.5] 탑   8     ☐   지난 80×7    ← 직전값 회색
 4   [75] 백오프  10    ☐   지난 75×10
 5   [75] 백오프  10    ☐
        └ 노력도(선택): 여유 · 2~3 남음 · 거의 한계 · 실패 · ✕
```

| 보강 | 현재 | 제안 |
|---|---|---|
| 직전 기록 표시 | 배너엔 todayTarget만 | **세트행에 직전 같은 세트값 회색 표시**("지난 80×8") — 진행 결정의 90% |
| 탑/백오프 구분 | 대표(최대)무게만 합산 | **세트행 [탑]/[백오프] 라벨** + 배너 "탑세트 기준" 표기 |
| 증량 프리필 | 다음 무게 자동 채움 | 유지 + "증량: 80→82.5kg, 반복 하단 리셋" 1줄(심각4 보강) |
| RIR 칩 | 수집·저장됨 | 선택형 유지 + 기본 OFF, 핵심 종목 마지막 세트에만 노출 |

### B-3. 머신 가변 종목(상업 헬스장)
가변 머신 배너는 무게 목표 대신 **"이 부위 볼륨 1세트 적립 + 적정 RIR"** 문구. 무게칸 옆 "머신 다름?" 토글(선택) → 그날 무게를 진행 판정에서 제외 표시(MVP는 라벨만).

# Q5 — RIR을 매번 받는 게 효율적인가

## 5-1. 결론 한 줄

**매번은 비효율적이다.** 근비대 진행 판정의 90%는 무게·반복·세트로 결정되고, RIR이 추가로 사는 정보는 "같은 수행이 쉬워졌나(0~3 RIR 게이트)"라는 한 비트뿐이다. **기본 비활성 + 탑세트/핵심 종목의 마지막 세트에만 선택적으로** 받는 게 정확도/마찰 균형의 최적점이다. 현재 코드 구조(인라인 칩 + null 폴백)와 일치.

## 5-2. 트레이드오프 — 4개 수집 정책

| 정책 | 정확도 기여 | 마찰 | 위험 |
|---|---|---|---|
| **매번(전 세트)** | 한계 효용 급감 | 높음 → 로깅 이탈 | 초보 RIR 부정확(Steele 2017) → 노이즈가 곱해짐 |
| **핵심 종목 마지막 세트만** | 높음(유효 반복 결정) | 낮음(종목당 1탭) | 거의 없음 |
| **선택형(원할 때만)** ← 확정 | 받았을 때만 보정 | 최저 | "안 받으면 게이트 헐거워짐"을 폴백으로 막아야 |
| **아예 안 받음** | 0 — 톤수 거품 못 잡음 | 0 | 정체·백오프서 "진짜 자극인가" 판별 불가 |

근거 3개: ① 근비대엔 0~3 RIR 게이트면 충분(연속값 불필요) — 현 `Effort` enum(EASY/MODERATE/HARD/FAILURE)이 옳은 해상도. ② 초보 RIR 부정확 → 레벨별 신뢰도 차등이 정답(초보=참고만, 중급+=게이트 반영). ③ 톤수 단독 판정의 함정을 막는 유일한 신호 — **받을 가치가 가장 큰 순간 = 탑세트+백오프의 백오프 세트, 정체 의심 종목**. 사용자가 탑세트+백오프 선호하므로 실익이 평균보다 크다.

## 5-3. 설계 — 선택형 RIR 4규칙

**규칙 1 — 기본 비활성, 종목 단위 옵트인**: `ExerciseGoal.effortTracking: boolean`(기본 false) 추가. core 역할 + 켠 종목만 칩 노출. 전역 강제 없음.

**규칙 2 — 받는 위치를 탑세트·백오프에 맞춤**:

| 세트 유형 | RIR 받나? | 이유 |
|---|---|---|
| 워밍업 | ✕ | 진행 무관 |
| 탑세트(대표 무게 마지막) | △ 선택 | 강도 축 |
| **백오프 마지막 세트** | ◎ 권장 | 볼륨 거품 판별 핵심(사례 B) |
| 중간 백오프 세트 | ✕ | 마지막 1개로 충분 |

→ 종목당 최대 1~2탭.

**규칙 3 — 안 받아도 무게·반복으로 판정(폴백이 1급 시민)**:
```
오늘 목표·증량 판정 = 무게·반복·세트(항상) + RIR(있을 때만 보정)
```
- RIR 미입력 → `two_sessions` 트리거로 판정(보수적 안전). RIR 입력 → `rpe`로 1세션 만에 허용/보류.
- **중요 가드**: 현 `rpe` 폴백은 null일 때 `single`(1세션)로 떨어진다(`ProgressionEngine.java:115`) → "RIR 안 주면 오히려 게이트가 헐거워지는" 역설. **폴백을 `two_sessions`로 교정**해야 "안 받아도 손해 없고 받으면 한 세션 빨라진다"는 올바른 인센티브.

**규칙 4 — 받았을 때만 작동하는 자기조절 보정(양방향)**:

| 신호 | 조건(RIR 있을 때만) | 엔진 동작 |
|---|---|---|
| 증량 가속 | 탑세트 EASY/MODERATE + 상단 달성 | 1세션 만에 READY_TO_INCREASE (현 구현) |
| **증량 보류** | 상단 달성했지만 마지막 세트 HARD/FAILURE | 증량 보류, "한 번 더 다지기"(현재 미구현, 심각5) |
| **볼륨 거품 경고** | 백오프 마지막 세트 EASY인데 톤수만 증가 | "톤수는 늘었지만 마지막 세트가 가벼웠어요"(사례 B 차단) |

→ 현재는 증량 방향만 본다. **양방향(가속+감속)**이 핵심.

# Q6 — 탑세트 + 백오프에서 점진적 과부하 계산·판단·추천

## 6-1. 핵심 원리 — 두 축을 분리해 판정한다

| | 탑세트 (고강도 1~2세트) | 백오프 (볼륨 3~N세트) |
|---|---|---|
| 목적 | **강도/근력 자극** (질·강도 축) | **볼륨 자극** (양 축) |
| 진행 판정 기준 | **무게·e1RM 추세**(저반복이라 e1RM 신뢰) | **반복·볼륨 채움**(이중진행 반복 우선) |
| 증량 신호 | 톱세트 RPE 여유 / e1RM 우상향 | 전 백오프 세트 반복 상단 도달 |
| 함정 | 매번 즉흥 신기록 욕심(가짜 정체) | 무게 즉흥 인하 → 톤수 거품(사례 B 강주의) |

핵심: **두 세트군을 데이터로 구분(세트 타입 태그)하지 않으면 둘이 한 덩어리로 뭉개진다.** 현 코드 `progressionReps`는 **대표(최대)무게 세트만** 합산하므로, 태그 없이 합치면 탑세트(100kg)만 잡히고 백오프(85kg)는 "가벼운 세트"로 통째 무시된다 → 백오프 볼륨 진행을 못 본다. 그래서 태그가 필수다.

## 6-2. 데이터 구분 — 세트 타입 태그 (스키마)

현 `SetType` = `NORMAL, WARMUP, DROP, FAILURE`(코드 확인). **`TOP_SET`, `BACKOFF` 2종 추가**가 Q6의 토대다. `effort`(RIR) 칸은 이미 존재 → 추가 스키마 없이 재사용.

```
SetType:  NORMAL | WARMUP | DROP | FAILURE | TOP_SET | BACKOFF   (← 2종 추가)
```

| 신규 헬퍼 | 정의 | 용도 |
|---|---|---|
| `topSets(r)` | setType=TOP_SET 작업 세트 | 강도축 판정 |
| `backoffSets(r)` | setType=BACKOFF 작업 세트 | 볼륨축 판정 |
| `topProgression(r)` | 톱세트 무게·e1RM | 톱 증량 게이트 |
| `backoffProgression(r)` | 백오프 대표무게 세트 reps 합 | 백오프 이중진행 게이트 |

태그 미지정(기존 NORMAL만 있는 종목)이면 현 동작 그대로(**하위호환**). 세트 타입은 한 번 지정하면 다음 세션 pre-fill.

## 6-3. 진행 판정 로직 — 두 게이트를 따로 돌린다

```
[탑세트 게이트 — 강도축]
 if 톱세트 RIR ≥ 2(쉬움)                    → 톱 증량 신호(무게↑)
 elif e1RM 4~6세션 추세 우상향 && 반복↑     → 진행 중(유지)
 elif 같은 무게 3세션 연속 톱 반복 비증가    → 톱 정체 → 백오프로 볼륨 보강 먼저
 (저반복이므로 e1RM 신뢰 — 단독 강등 금지)

[백오프 게이트 — 볼륨축]  (이중진행 · 반복 우선)
 if 전 백오프 세트가 반복 상단 도달   → 백오프 무게 +1단(마이크로) · 반복 하단 리셋
 else                                → 같은 백오프 무게, 총 반복 +1~2 (오늘 목표)
 가드: targetTotalReps ≤ prevTotal + 2 (급점프 금지)
```

증량을 **따로** 트리거: 톱은 무게로, 백오프는 반복으로. **둘을 같은 날 동시에 안 올린다**(2주 내 한쪽씩 권장).

## 6-4. 구체 예시 — 직전 = 탑 100×5 + 백오프 85×8/8/8 (백오프 범위 8~10 가정)

| # | 직전 톱 | 직전 백오프 | 톱 판정 | 백오프 판정 | 다음 처방 |
|---|---|---|---|---|---|
| 1 | 100×5 @2~3RIR(쉬움) | 85×8/8/8(상단 미달) | 여유 → 증량 | 상단 미달 → 반복 더 | **탑 102.5×5** · 백오프 85kg 총 +1~2(85×9/8/8) |
| 2 | 100×5 @0~1RIR(빠듯) | 85×10/10/10(전 세트 상단) | 빠듯 → 유지 | 상단 → 증량 | 탑 100×5 유지 · **백오프 87.5×8/8/8**(반복 리셋) |
| 3 | 100×5 @1RIR | 85×10/10/10 + 톱도 여유 | 여유 → 증량 | 상단 → 증량 | 둘 다 한 단계(탑 102.5 / 백오프 87.5) — 단 2주 안엔 한쪽씩 권장 |
| 4 | 100×4(반복 미달) | 85×8/7/6(드롭오프 큼) | 미달 → 점검 | 하락 → 회복 점검 | 탑 100×5 재도전 · 백오프 85kg 유지, 세트간 휴식↑(8/6/4 급락=회복 부족) |
| 5 | 100×5 | **즉흥 80×9/9/9**(무게 임의 인하) | — | **비교 보류** | "백오프 무게가 달라 직접 비교 안 함" — 톤수 거품·추적선 파괴. 백오프 무게 **고정 규칙화** 안내 |

**케이스 1을 풀어쓰면(가장 흔함)**: 탑 100×5(RPE 8 = 1~2 RIR) → "RPE 8쯤이면 다음에 증량" → **탑 102.5kg×5**(하체 +2.5). 백오프 8/8/8은 상단(10) 미충족 → 이중진행 반복 우선 → **무게 유지, 백오프 총 +1~2회**(85×9/8/8). 화면:
```
스쿼트
 탑    102.5kg × 5     (지난 100×5가 여유 → 한 단 올림)
 백오프 85kg, 합 25~26회 노리기 (지난 24 → +1~2)
 성공조건: 톱 5회 + 백오프 마지막 세트 1~2개 남기는 느낌
```

## 6-5. 백오프 증량 환산 규칙 (사례 B "다음엔")

```
85×8/8/8 @2RIR  →  85×9/9/8 @1RIR  →  85×10/10/10  →  87.5×8/8/8 (리셋)
```
- 즉흥 인하 금지(케이스 5). 백오프 무게 = 탑세트의 일정 비율(예 85%)로 **자동 고정 제안** → 톱 올리면 백오프도 같은 비율로 따라 올림(Madcow·FSL 패턴).
- 백오프 RIR(선택형)이 헐거워지면(2→4RIR) 톤수가 늘어도 거품 — RIR ON일 때만 경고.

## 6-6. 정체·비교 보류
- 백오프 세트수가 직전과 2 이상 차이 → `COMPARISON_DEFERRED`("조건 달라 비교 안 함", "하락" 표현 절대 금지). 케이스 5 즉흥 인하도 이 보류로.
- 톱 정체는 단발 아닌 추세로만(STALL_REVIEW 게이팅). 톱 막히면 **백오프 볼륨 보강 먼저**(세트 추가) → 재정체 시 5~10%↓.

# Q7 — AI가 효과가 있을까, 어떤 방식이어야 하나

## 7-1. 결론 한 줄

**처방(숫자)은 100% 결정론 엔진이 단일 소스로 내고, AI는 그 처방의 "왜"를 설명·번역·동기부여하는 보조로만 쓸 때 효과가 있다.** AI가 처방 숫자를 새로 만들면(환각 처방) 일관성·신뢰가 무너져 순효과가 음(−)이 된다. 현재 코드는 이미 이 경계("계산은 코드, 해석만 LLM" + 출력 검증기 `AiReportValidator`)를 채택했고, 옳은 아키텍처다. 남은 일은 **경계를 더 단단히 + 적용 범위 확대**.

## 7-2. 효과 있는 패턴 vs 위험한 패턴

| | 효과 있음 (✅ 채택) | 위험 (❌ 금지) |
|---|---|---|
| 처방 숫자 | 엔진의 `todayTarget`/`nextTarget` **인용**만 | AI가 "다음엔 84kg 9회" 식 **창작** |
| 정체 판정 | 엔진 `stalling` 결과를 **해석·위로** | AI가 "정체인 것 같다" 독립 판단 |
| 일관성 | 같은 입력 → 같은 처방 | 같은 입력 → 매번 다른 숫자 |
| 검증 | 출력 수치를 입력 stats와 대조 → 환각 차단 | 검증 없이 LLM 출력 노출 |

왜 결정론 엔진이 단일 소스인가: ① **일관성** — 같은 80kg 16회에 오늘도 내일도 17회를 줘야 신뢰가 쌓인다. ② **환각 위험** — 처방은 사용자가 그대로 따르는 행동 지시. LLM이 "94kg 12회" 같은 비현실적 숫자를 만들면 부상·좌절로 직결(엔진은 급점프 가드로 막지만 LLM은 못 막음). ③ **이미 검증된 설계** — 백엔드 원칙 "수치는 코드가 확정해 LLM에 사실로 전달, LLM은 재계산 금지".

## 7-3. AI의 5개 허용 역할 (처방 숫자 안 만듦)

| # | 역할 | 입력(엔진 fact) | AI가 하는 일 | 환각 가드 |
|---|---|---|---|---|
| 1 | "왜" 설명 | stage, todayTarget | "지난주 16회 채웠으니 오늘은 17회만 — 무게 그대로 반복 쌓는 단계" | 숫자는 엔진 값 인용만 |
| 2 | 정체/회복 해석 | 엔진 stalling + RIR 추세 | "3주째 무게 정체인데 마지막 세트가 점점 힘들어졌어요 — 회복부터" | 판정은 엔진, AI는 맥락만 |
| 3 | 메모/세션 요약 | 세트 데이터 + 메모 | "오늘 가슴 14세트, 벤치 신기록. 어깨 뻐근하다 적으셨네요" | 세트 수는 stats 값만 |
| 4 | 자유 질문답변 | 기간 stats로 스코프 제한 | "왜 백오프를 하나요?" → 원리 설명 | 기간 밖 "모른다", 숫자 창작 금지 |
| 5 | 동기부여 | 출석·streak·개선 | "벌써 2번 나오셨네요, 이 페이스면 충분" | 비하·다그침 금지 |

→ 전부 **"엔진이 이미 정한 것을 사람 말로 번역"**. 새 숫자가 끼어들 자리가 없다.

## 7-4. 환각 처방을 막는 3중 방어 (현황 + 보강)

```
1차: 프롬프트 계약 — "새 숫자 만들지 마라. facts/엔진 값만 인용"        ✅ 있음
2차: 입력에 엔진 처방을 fact로 주입 — todayTarget/nextTarget을 facts에   부분(종목코치엔 있음, 채팅 보강)
3차: 출력 검증 — AiReportValidator가 처방·헤드라인 수치를 입력과 대조      ✅ 리포트만(채팅·코치로 확장)
```

**보강**: 처방성 답변(다음 무게/반복) 가능한 채팅·종목코치에 "엔진 처방 fact 주입"을 의무화하고 AI는 인용만. "다음 무게?" 질문은 LLM이 답하지 말고 **엔진 `nextTarget`을 카드로 띄우고 AI는 이유만 설명**하는 라우팅.

## 7-5. 비용 대비 가치
AI는 **"처방을 만드는 비싼 두뇌"가 아니라 "엔진 결과를 사람 말로 옮기는 싼 통역"**으로 쓸 때 가치가 양(+)이다. 처방 숫자 생성은 환각·일관성 붕괴로 음(−). 주간 리포트·정체 해석이 가치 최고(◎), 메모요약·자유질문·동기는 보조(○).

# Q8 — 종목별 분석 화면

# Q9 — 종목별 진행 룰을 데이터로 표현·판정 (목표 설정 모델)

## 9-1. 현 데이터 모델의 한계
현 `ExerciseGoal`은 룰을 `ruleType(4종) + repRangeMin/Max + targetSets + increment`로만 표현 → **단일 "더블 프로그레션"**만 가능. 사용자 선호 **탑세트+백오프**나 **총 반복 목표**는 표현 불가. 핵심: **`progressionRule` enum 추가 + 룰별 증량 트리거 차등**.

## 9-2. 룰 종류 카탈로그 (데이터 표현)

| progressionRule | 설명 | 파라미터 | 증량 트리거 | 적합 종목 |
|---|---|---|---|---|
| **DOUBLE_PROGRESSION** (기본) | 같은 무게로 범위 상단 채우면 증량 | repMin, repMax, sets | 모든 본세트 ≥ repMax (트리거 세션수만큼) | 대부분·고립·머신 |
| **TOP_SET_BACKOFF** (사용자 선호) | 앞 N세트 탑(무게), 뒤 M세트 백오프(반복) | topSets, topReps, topWeight, backoffSets, backoffPct, backoffRepMax | 탑 무게 달성 + 백오프 반복 상단 | 스쿼트·벤치·데드 본운동 |
| **TOTAL_REP_TARGET** | 세트 구성 자유, 총 반복만 목표 | totalRepTarget, weight | 목표 무게서 총 반복 달성 | 맨몸·펌프 종목 |
| **RIR_AUTOREG** | 무게 추적 포기, 같은 RIR 유지 | targetRIR, repRange | 같은 RIR서 반복↑ | VARIABLE 머신 |

## 9-3. 룰 → 판정 의사코드

**(A) DOUBLE_PROGRESSION — 현 엔진 그대로**
```
증량신호 = metTop(prev, repMax, sets) AND 트리거충족
오늘목표 = 미충족 ? "{w}kg 총 {prevTotal+1~2}회" : "{w+inc}kg {repMin}회부터"
```

**(B) TOP_SET_BACKOFF — 사용자 선호, 신규 (탑세트 무게·백오프 반복 분리 추적)**
> 사례 B 결론: "백오프 무게를 미리 고정해 매번 동일하게. 즉흥 백오프는 톤수만 부풀린다."
```
판정(prev, 룰):
  topSets    = 무게 == topWeight 인 작업세트            // 가장 무거운 세트군 = 탑세트
  backoffSets= 무게 == round(topWeight * backoffPct)   // 예: 90%

  topReady     = topSets 모두 reps >= topReps           // 무게 기준 진행
  backoffReady = backoffSets 모두 reps >= backoffRepMax // 반복 기준(더블 프로그레션)

  if topReady:     탑 증량 → topWeight += inc, topReps 리셋
  elif 탑 reps 미달: "탑세트 {topWeight}kg {topReps}회 한 번 더"
  if backoffReady:  백오프 증량 → backoffWeight += inc(절반), 반복 하단 리셋
  else:             "백오프 {backoffWeight}kg 총 반복 +1~2"

  추적선: 탑세트 = 무게 추세(e1RM 보조), 백오프 = 반복/RIR 추세
```

**예시 — 스쿼트 탑세트+백오프 (5세트):**

| 세션 | 탑세트(2세트, 무게축) | 백오프(3세트, 반복축) | 엔진 판정 | 오늘 처방 |
|---|---|---|---|---|
| 1 | 100kg 5/5 | 90kg 8/8/7 | baseline | — |
| 2 | 100kg 5/5 | 90kg 9/9/8 | 백오프 반복↑ → IMPROVED | "백오프 90kg 총 27회(어제 26)" |
| 3 | 100kg 5/5 | 90kg 10/10/10 | 백오프 상단 → 증량 | "백오프 92.5kg 8회부터 / 탑 100kg 유지" |
| 4 | 100kg 6/5 | 92.5kg 8/8/8 | 탑 reps↑(6/6면 증량) | "탑 100kg 6/6 채우면 102.5kg" |
| 5 | 100kg 6/6 | 92.5kg 9/8/8 | 탑 상단 → 증량 | "탑 102.5kg 5회부터" |

**(C) TOTAL_REP_TARGET — 세트 구성 자유**
```
done = Σ 작업세트 reps (목표무게)
if done >= totalRepTarget: 증량(무게↑, target 유지)
else: "총 {done+2}회 (목표 {totalRepTarget})"   // 세트 수 자유
```
맨몸·펌프 종목 적합. 8/8/8/6/6 같은 자연 드롭오프를 정상으로 흡수.

**(D) RIR_AUTOREG — VARIABLE 머신**
```
무게 비교 불가 → done 만 봄
if RIR입력 있고 같은RIR서 cur반복 > prev반복: IMPROVED
else: "부위 볼륨 1세트 적립 · 실패 1~3회 전까지"
```

## 9-4. 룰별 증량 트리거 요약

| 룰 | 무엇을 올리나(1차) | 증량 게이트 | 트리거(세션) | RIR 역할 |
|---|---|---|---|---|
| DOUBLE_PROGRESSION | 반복 → 무게 | 전 세트 상단 | single/two_sessions/rpe | rpe면 여유 있을 때만 |
| TOP_SET_BACKOFF | 백오프 반복 / 탑 무게(분리) | 탑 reps 충족 OR 백오프 상단 | two_sessions 권장 | 탑세트에 우선 |
| TOTAL_REP_TARGET | 총 반복 → 무게 | 총 반복 목표 달성 | single | 선택 |
| RIR_AUTOREG | 같은 RIR 반복 | (무게 게이트 없음) | — | 필수(없으면 볼륨만) |

## 9-5. 입력 부담 줄이는 기본값 + 수정 방식
`OverloadService.classifyRuleType/classifyRole`이 이미 장비·부위로 ruleType·role을 자동 분류한다. 여기에 `progressionRule`·`anchorTier` 자동 기본값을 얹어 **사용자는 0개 입력으로 시작, 원할 때만 수정**:
```
종목 등록 시 자동:
  ruleType        = classifyRuleType(장비, 부위)        // 기존
  anchorTier      = ANCHOR if 프리웨이트/체중 else MAIN_MACHINE  // 신규
  progressionRule = barbell_main → (사용자가 켜면) TOP_SET_BACKOFF, 아니면 DOUBLE_PROGRESSION
                    machine/isolation → DOUBLE_PROGRESSION
                    bodyweight → TOTAL_REP_TARGET
  repRange/sets   = repRange(kind, goalType)            // 기존 단일 소스
```
- 기본값으로 바로 운동 가능. 룰 변경은 종목 상세 한 칸(드롭다운).
- TOP_SET_BACKOFF 선택 시 추가로 받는 건 `topSets·topReps·backoffPct` 3개뿐. 첫 기록에서 탑/백오프 무게 자동 추론.
- `is_overridden` 플래그(기존) 유지: 사용자가 한 번 손대면 자동 재분류가 덮어쓰지 않음.
