# 진행 방식 9종 — 앱 적용 관점 분석

> 목적: 점진적 과부하의 9개 진행 방식을 "이 앱에 실제로 어떻게 적용되는가" 관점으로 분석한다.
> 배경 근거: `docs/PROGRESSIVE_OVERLOAD_RESEARCH.md`(논문·전문가·시스템 교차검증 + 4인 패널 결론)를 재활용.
> 검증 기준: 실제 엔진 코드(`gymtracker-backend/.../overload/ProgressionEngine.java`, `OverloadService.java`)와 저장 데이터(`GoalSetting`, `ExerciseGoal`)를 직접 읽고 작성.
> 타겟: 본인 1명(Strong 헤비유저, 중급+). 입력 최소화·헬스장 3초 이해가 철칙.
> 작성일: 2026-06-29

---

## 0. 먼저 — 이 앱이 실제로 가진 데이터 (분석의 전제)

각 방식의 "앱 필요 입력값 / 계산값"이 현실적인지 판단하려면 앱이 무엇을 이미 들고 있는지 알아야 한다. 코드에서 확인한 사실:

**온보딩에서 1회 저장되는 값** (`GoalSetting`):
- `goalType` (hypertrophy|strength|fatloss|endurance)
- `weeklyFrequency` (주당 빈도)
- `incUpper` / `incLower` (상체·하체 증량 단위 kg)
- `progressionTrigger` (single|two_sessions|rpe) — 기본 two_sessions

**종목별로 저장되는 값** (`ExerciseGoal`):
- `ruleType` (barbell_main|machine_cable|isolation|bodyweight) — 장비·부위로 자동 분류
- `role` (core|support|log_only)
- `targetReps`, `targetSets`(기본 3), `repRangeMin/Max`, `increment`, `currentValue`, `nextTarget`

**매 세트 기록되는 값** (워크아웃):
- weight, reps (워밍업 플래그)
- **RPE/RIR는 타입에만 존재하고 실제 저장 안 됨** — `SetResult.rpe`는 프론트 타입에 있으나 엔진은 effort 데이터를 받지 못함(`readyToIncrease`의 rpe 분기가 single로 폴백).

**엔진이 실제로 계산하는 것**:
- 진행 기준 반복수 = 대표(최대) 무게 세트들의 반복수 합 (맨몸은 전체 합)
- 증량 준비 판정 = `repRangeMax`를 `targetSets`만큼 채웠는가 + 트리거(1세션/2세션)
- 정체 = 같은 무게 3회 연속 반복수 비증가
- 증량폭 = 머신/고립은 절반(최소 0.5kg), 그 외는 사용자 increment 그대로

이 전제가 "MVP 포함 여부"를 가른다. **앱이 안 가진 입력(RPE, 주간 볼륨 추적 UI, 메소사이클 주차)을 요구하는 방식은 MVP에서 빠진다.**

## 1. Linear Progression (선형 진행)

- **개념**: 매 세션 무게를 고정 단위만큼 올린다. 반복수는 고정(예 5×5), 무게만 단조 증가.
- **적합 사용자**: 초급. 회복이 빨라 세션마다 적응이 일어나는 단계.
- **적합 운동유형**: BARBELL_COMPOUND (스쿼트·벤치·데드 등 큰 운동에서만 지속).
- **앱 필요 입력값**: increment(증량 단위), targetReps×targetSets, currentValue. — 전부 이미 있음.
- **앱 계산값**: nextTarget = current + increment. (`computeNextTarget`가 정확히 이것)
- **사용자에게 보여줄 목표**: "62.5kg 5회 × 3세트" → 성공하면 "65kg".
- **장점**: 가장 단순·명확. 3초 이해. 초급 구간 진행 속도 최강.
- **단점**: 중급+에선 금방 막힘(매 세션 증량 불가능). 막히면 잦은 실패·디로드.
- **내 앱 기본값 적합성**: **낮음.** 타겟이 중급+ 1명이라 매 세션 선형 증량은 비현실적. 단, `single` 트리거 + 무게 고정 흐름이 사실상 "성공 시 선형 증량" 형태로 일부 살아있음.
- **MVP 포함 여부**: **부분 포함.** 독립 모드로는 불필요. `single` 트리거 선택 시 결과적으로 선형처럼 동작.

## 2. Double Progression (이중 진행)

- **개념**: 무게 고정 → 반복 범위(예 8~12) 상단까지 반복을 늘림 → 모든 세트가 상단 도달하면 무게 올리고 하단으로 리셋. 반복 먼저, 무게 나중.
- **적합 사용자**: 초급~고급 전반. 특히 중급의 표준.
- **적합 운동유형**: MACHINE_OR_CABLE, ISOLATION, 비대 목적 BARBELL_COMPOUND. 거의 전부.
- **앱 필요 입력값**: repRangeMin/Max, targetSets, increment, currentValue. — 전부 있음.
- **앱 계산값**: "repRangeMax를 targetSets만큼 달성했는가"로 증량 준비 판정 → 달성 시 nextTarget 제시. (엔진 `readyToIncrease` + `metTop`가 정확히 이 로직)
- **사용자에게 보여줄 목표**: "60kg 8~12회 × 3세트, 전 세트 12회 채우면 증량".
- **장점**: 무게/반복 두 축을 다 써서 막힘이 적음. 모든 분류에 자연스럽게 일반화. 입력은 무게·반복뿐(앱이 이미 받는 것).
- **단점**: "전 세트 상단 도달" 판정이 세트 구성 변동에 민감(엔진이 세트수 유사성으로 보정 중).
- **내 앱 기본값 적합성**: **최상.** 엔진의 BUILD_REPS → READY_TO_INCREASE 흐름 자체가 double progression. 연구 문서도 "정통 double progression, Hevy Trainer와 일치"로 평가.
- **MVP 포함 여부**: **포함 (핵심 기본값).**

## 3. Rep Range Progression (반복 범위 진행)

- **개념**: 분류별로 적절한 반복 범위를 정하고, 그 안에서 반복을 채우는 데 집중. (Double Progression의 "범위" 절반에 해당하는 개념적 토대)
- **적합 사용자**: 전 레벨. 특히 근비대 목적.
- **적합 운동유형**: 전부. 다만 범위 값이 분류마다 달라야 함.
- **앱 필요 입력값**: 분류(ruleType), goalType. — 자동 결정됨.
- **앱 계산값**: `ProgressionEngine.repRange()` 단일 소스 — 비대 바벨 6~12 / 머신 8~12 / 고립 8~15 / 맨몸 5~12, 근력 바벨 3~6.
- **사용자에게 보여줄 목표**: "이 종목 권장 범위 8~15회".
- **장점**: 종목 성격에 맞는 범위를 자동 부여 → 사용자가 직접 안 정해도 됨(입력 최소화). 고립을 8~15로 낮춰 증량 출구 확보(최근 변경).
- **단점**: 단독으로는 "진행"을 만들지 못함 — 증량/볼륨 같은 진행 규칙과 결합해야 함.
- **내 앱 기본값 적합성**: **높음(기반 인프라).** 그 자체가 독립 방식이라기보다 Double Progression의 입력 토대. 단일 소스화는 이미 완료.
- **MVP 포함 여부**: **포함 (Double Progression의 토대로).**

---

## 4. Total Rep Progression (총 반복수 진행)

- **개념**: 세트별 반복을 고정하지 않고 "오늘 총 N회"를 목표로. 무게 고정, 세트 분배 자유, 총합만 +1~2씩 증가.
- **적합 사용자**: 중급+, 특히 컨디션 변동이 큰 날에도 목표 달성 가능해야 하는 사용자.
- **적합 운동유형**: MACHINE_OR_CABLE, ISOLATION, BODYWEIGHT. (대표 무게가 흔들리는 종목에 강함)
- **앱 필요 입력값**: currentValue, 직전 총 반복수. — 기록에서 계산.
- **앱 계산값**: target = 직전 진행기준 반복수 + bump(1~2). 엔진 BUILD_REPS가 정확히 "총 X~X+1회" 제시.
- **사용자에게 보여줄 목표**: "60kg 총 25~26회" (세트 나눔은 자유).
- **장점**: 근비대의 진짜 기준(유효 볼륨)에 가장 직접적으로 부합. 컨디션 나쁜 날도 세트를 더 쪼개 달성 가능 → 좌절 적음. 입력은 weight·reps뿐.
- **단점**: "총합만 본다"는 게 폼 붕괴를 부추길 수 있음(렙당 질 저하). 무게 진행 신호가 약함.
- **내 앱 기본값 적합성**: **높음.** 엔진이 무게 기반 종목의 BUILD_REPS를 이미 "총 반복수 +1~2"로 처방. 근비대 관점에서 올바른 방향(연구 문서 4번 결론).
- **MVP 포함 여부**: **포함 (BUILD_REPS 단계의 처방 형태로).**

## 5. Top Set + Back-off (탑세트 + 백오프)

- **개념**: 무겁고 적은 반복의 탑세트 1개(또는 소수) → 무게를 낮춘 백오프 세트 여러 개로 볼륨 확보. 강도와 볼륨을 한 세션에 분리.
- **적합 사용자**: 중급~고급. 강도와 볼륨을 동시에 노리는 단계.
- **적합 운동유형**: BARBELL_COMPOUND (5/3/1·nSuns의 구조).
- **앱 필요 입력값**: 탑세트 목표 강도(% 또는 절대무게), 백오프 비율, 세트수. — **현재 미보유.** 앱은 세트별 무게를 자유 입력받을 뿐 "탑/백오프 역할"을 구분하지 않음.
- **앱 계산값**: 탑세트 1RM 추정 → 백오프 무게(예 탑의 85%) 자동 산출. — 미구현.
- **사용자에게 보여줄 목표**: "탑 80kg×5, 백오프 65kg×8×3".
- **장점**: 강도·볼륨을 한 운동에 통합. 상급 바벨 진행에 강력.
- **단점**: 새 입력(역할 구분·% 기준)이 필요해 입력 최소화 철칙에 위배. 엔진의 "대표 무게 세트만 비교" 로직과 충돌(탑 1세트 vs 백오프 다세트를 어떻게 비교?).
- **내 앱 기본값 적합성**: **낮음.** 구조가 현재 데이터 모델(역할 없는 평면 세트)과 안 맞음.
- **MVP 포함 여부**: **제외.** 향후 고급 프로그램 모드의 후보.

## 6. RIR/RPE 기반 진행 (자기조절)

- **개념**: 세트의 노력도(RIR=남은 반복, RPE=체감 난이도)를 게이트로 사용. 여유(예 RPE≤7, ≥3 RIR)가 있으면 증량, 없으면 유지.
- **적합 사용자**: 중급+ (초급은 RIR 추정이 부정확 — Steele 2017). RIR을 입력할 의지가 있는 사용자.
- **적합 운동유형**: 전부.
- **앱 필요 입력값**: 세트별 RPE/RIR. — **타입엔 있으나 실제 저장 안 됨.** 엔진의 rpe 트리거는 single로 폴백 중.
- **앱 계산값**: 마지막 세트 RPE ≤ 임계면 증량 준비. (`evaluateProgression`의 rpe 분기에 로직은 있으나 데이터 부재로 비활성)
- **사용자에게 보여줄 목표**: "마지막 세트 2~3개 남기는 느낌이면 다음에 증량".
- **장점**: %1RM 대비 최소 동등하면서 컨디션 자동 반영. 과/저훈련 방지. 처방 정확도 향상.
- **단점**: 매 세트 RPE 입력 = 입력 부담(철칙 위배 소지). 초급·실패에서 먼 구간일수록 부정확. 현재 저장 파이프라인 없음.
- **내 앱 기본값 적합성**: **중간.** 트리거 옵션으로는 가치 있으나, **강제 입력은 금지**(4인 패널: 선택 입력 + 보조 반영만). 현재는 "마지막 세트 여유" 카피로만 노출.
- **MVP 포함 여부**: **부분 포함.** 트리거 선택지로는 존재. 실제 RPE 데이터 연동은 후속(선택 입력으로).

## 7. Volume Progression (볼륨 진행)

- **개념**: 무게·반복이 아니라 주간 유효 세트 수를 점증(예 부위당 12 → 14 → 16세트). 메소사이클에 걸쳐 MEV→MRV로 세트 추가.
- **적합 사용자**: 중급~고급. 무게 증량 여지가 줄어든 단계.
- **적합 운동유형**: MACHINE_OR_CABLE, ISOLATION (무게보다 세트 추가가 자연스러운 종목).
- **앱 필요 입력값**: 주간 세트 추적, 메소사이클 주차, 부위별 목표 세트. — 부위별 주간 세트는 이미 집계됨(`weeklySetsByMuscle`, `targetSetsForPart`), **메소사이클 주차·자동 점증은 미보유.**
- **앱 계산값**: 정체 시 "세트 추가" 제안. 부위별 주간 세트 vs 권장(12~20) 진행도 표시.
- **사용자에게 보여줄 목표**: "가슴 이번 주 10/12세트 — 오늘 더 넣으면 채워짐", 정체 종목엔 "세트 추가 검토".
- **장점**: 근비대의 1차 기준(유효 볼륨)에 직접 부합. 고립·머신 증량 막힘의 진짜 출구.
- **단점**: MEV/MRV는 검증 모델 아닌 휴리스틱(과신 금지). 풀 구현은 비용 큼. 주차 추적 UI 필요.
- **내 앱 기본값 적합성**: **중간.** "정체 시 세트 추가 출구" + "부위별 주간 세트 노출"까지만 채택. 자동 메소사이클 점증은 보류(4인 패널 만장일치).
- **MVP 포함 여부**: **부분 포함.** 세트추가 출구 + 부위별 세트 진행도 노출만. 자동 볼륨 점증은 제외.

## 8. Step Loading (계단식 부하)

- **개념**: 매번 올리지 않고 며칠/몇 주 무게를 유지해 적응시킨 뒤 한 번에 올림. "유지(consolidate) → 증량 → 다시 유지" 계단 패턴.
- **적합 사용자**: 중급~고급 (세션 단위 증량이 안 되는 단계).
- **적합 운동유형**: BARBELL_COMPOUND, MACHINE_OR_CABLE.
- **앱 필요 입력값**: 동일 무게 유지 세션 수, 증량 후 적응 단계 표시. — `two_sessions` 트리거가 "2세션 유지 후 증량"으로 이미 계단 1칸을 구현.
- **앱 계산값**: 2세션 연속 달성 → 증량 준비. 증량 직후 CONSOLIDATE/INCREASE_LOAD(타입 존재, 현재 전이 미구현).
- **사용자에게 보여줄 목표**: "이 무게로 한 번 더 안정시키고 다음에 증량".
- **장점**: 무리한 매세션 증량을 막아 폼·관절 보호. 중급+ 현실에 맞음. 기본 트리거(two_sessions)와 자연스럽게 결합.
- **단점**: 증량 직후 "적응 단계"(CONSOLIDATE)가 타입만 있고 전이 안 됨 → 곧바로 BUILD_REPS 복귀(계단의 평평한 칸이 약함).
- **내 앱 기본값 적합성**: **높음.** 기본 트리거 two_sessions가 이미 계단식의 핵심. 증량 후 1세션 하단 재확립(INCREASE_LOAD) 복구는 가치 있는 보강(연구 P0-3).
- **MVP 포함 여부**: **포함 (two_sessions 트리거 형태로).** 증량 직후 적응 스테이지 전이는 후속 보강.

---

## 9. Deload / Reset (디로드/리셋)

- **개념**: 정체·피로 누적 시 무게·볼륨을 일시 감량(예 무게 50~70%, 볼륨 30~50%)하고 재시작. 회복·지속성 도구.
- **적합 사용자**: 전 레벨(고급일수록 정기화). 정체·피로 신호가 뜬 사용자.
- **적합 운동유형**: 전부, 특히 BARBELL_COMPOUND(누적 피로 큼).
- **앱 필요 입력값**: 정체 판정(같은 무게 3회 비증가) — 이미 계산됨(`stalling`).
- **앱 계산값**: STALL_REVIEW 진입 → "세트 추가 또는 5~10% 감량 후 재시작" 제안.
- **사용자에게 보여줄 목표**: "3회 변화 없음 — 세트 추가하거나 5~10% 낮춰 재시작".
- **장점**: 정체 탈출·부상 예방·동기 유지. StrongLifts 3세션→감량 기준과 일치.
- **단점**: 디로드는 근비대를 촉진하지 않음(Coleman 2024) → "성장 재민감화" 카피 금지. 세션 단위 정체 판정은 정상 변동을 오판할 수 있음.
- **내 앱 기본값 적합성**: **높음(단, 포지셔닝 주의).** 정체 처리는 필수. 단 1차 정체에 바로 감량하지 말고 "세트 추가 먼저 → 재차 정체면 감량"(GZCL식 2단계)이 UX·근거 모두에서 우수.
- **MVP 포함 여부**: **포함 (정체 안전망으로).** 2단계 정체 처리가 권장 보강.

## 운동 유형별 기본 진행 방식 매핑 (필수 표)

각 분류에서 "기본으로 돌릴 방식"과 "정체 시 출구". 엔진 코드의 실제 분기와 일치하게 구성.

| 운동 유형 | 권장 반복 범위 | **1차 기본 방식** | 보조/출구 방식 | 증량폭 | 정체 시 |
|---|---|---|---|---|---|
| **BARBELL_COMPOUND** | 비대 6~12 / 근력 3~6 | **Double Progression** (+ Step Loading: two_sessions 트리거) | 근력 목표면 선형 성향↑ | 상체 +2.5kg / 하체 +5kg (사용자 increment 그대로) | 세트 추가 → 재차 막히면 5~10% Deload |
| **MACHINE_OR_CABLE** | 8~12 | **Double Progression** | Volume Progression(세트 추가), Total Rep | 마이크로로딩 = increment 절반(최소 0.5kg) | **세트 추가 우선** → Deload |
| **ISOLATION** | 8~15 | **Double Progression + Total Rep** | Volume Progression(세트 추가) | 마이크로로딩 = increment 절반(최소 0.5kg) | 세트 추가 우선 |
| **BODYWEIGHT** | 5~12 (풀업/딥스 임계 후 가중) | **Total Rep Progression** (총 반복수 +1~2) | 임계 도달 후 가중(+2.5~5kg)로 Double Progression 전환 | 반복 → 임계 후 무게 | 목표 미달 지속 시 HOLD |

핵심: **모든 분류의 공통 1차 엔진은 Double Progression**이며, 분류 차이는 (1) 반복 범위, (2) 증량폭(마이크로로딩 여부), (3) 정체 출구(머신/고립은 세트 추가 우선)에서만 갈린다. 이것이 엔진이 이미 구현한 구조와 정확히 일치한다.

## 결론 — 이 앱의 기본 진행 방식 추천 + 근거

**기본 진행 방식 = Double Progression (분류별 반복 범위 + 분류별 증량폭으로 파라미터화) + Step Loading(2세션 트리거) + 2단계 정체 안전망.**

근거:

1. **데이터가 이미 맞다.** Double Progression이 요구하는 입력은 weight·reps·범위·세트수뿐이고, 앱은 이걸 전부 갖고 있다. Top Set/Back-off(역할 구분 필요)나 RIR 진행(RPE 저장 필요)은 새 입력을 강요해 입력 최소화 철칙에 위배된다.

2. **하나의 엔진으로 4분류를 다 덮는다.** Double Progression을 (반복 범위, 증량폭, 정체 출구) 3개 파라미터로 분기하면 바벨·머신·고립·맨몸이 같은 코드 경로로 처리된다 — 실제 엔진이 그렇게 짜여 있다. 방식을 종목마다 따로 두지 않아 UX가 단순하다(헬스장 3초 이해).

3. **중급 1명 타겟에 정확.** 매 세션 선형은 비현실적이고, 메소사이클·MRV·자동 디로드는 검증 약한 휴리스틱이라 과한 복잡도다. 2세션 연속 달성(Step Loading) 트리거가 중급의 현실적 증량 속도와 맞는다(Helms·Rippetoe).

4. **근비대 기준에 부합.** 무게가 아니라 "유효 볼륨"이 1차 동인이므로, 무게 고정 + 총 반복수/세트 점증(Total Rep·Volume)을 Double Progression의 자연스러운 부분으로 흡수한다. 고립·머신은 마이크로로딩 + 세트 추가 출구로 막힘을 푼다.

5. **정체는 부드럽게.** Deload는 성장 촉진이 아니라 안전망이므로 "성장 재민감화" 카피를 쓰지 않고, 1차 정체엔 세트 추가를 먼저 제안한 뒤 재차 막히면 5~10% 감량한다.

MVP 채택: **Double Progression(핵심), Rep Range/Total Rep(그 토대·처방 형태), Step Loading(트리거), Volume Progression(세트추가 출구만), Deload(정체 안전망).**
MVP 제외: **Linear(독립 모드 불필요·single 트리거로 흡수), Top Set+Back-off(데이터 모델 불일치), RIR 강제 진행(선택 입력 보조로만).**

---

### 핵심 결론 8줄

1. 이 앱의 기본 진행 방식은 Double Progression이며, 엔진이 이미 그 구조로 짜여 있다.
2. 4개 운동 유형은 별도 방식이 아니라 같은 Double Progression을 (반복 범위·증량폭·정체 출구) 3파라미터로 분기한 것이다.
3. 바벨은 Double+Step Loading, 머신·고립은 Double+마이크로로딩+세트추가, 맨몸은 Total Rep(임계 후 가중)이 기본이다.
4. 기본 증량 트리거는 2세션 연속(Step Loading)이 중급 타겟에 맞고, single은 사실상 선형을 흡수한다.
5. Total Rep·Volume Progression은 독립 기능이 아니라 무게 고정 시 "총 반복수/세트 점증"으로 Double Progression에 흡수된다.
6. Top Set+Back-off, RIR 강제 진행은 새 입력을 요구해 입력 최소화 철칙에 위배되므로 MVP에서 제외(선택·후속).
7. Deload는 성장 촉진이 아닌 회복 안전망이라 "재민감화" 카피를 쓰지 말고, 1차 정체엔 세트 추가를 먼저 제안한다.
8. 결론: Double Progression(분류 파라미터화) + Step Loading 트리거 + 2단계 정체 안전망이 이 앱의 기본 진행 방식이다.
