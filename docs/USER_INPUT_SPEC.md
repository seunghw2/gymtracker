# GymTracker 사용자 입력 설계 (USER_INPUT_SPEC)

> 작성: User Input Design Agent. 근거 = `docs/APP_AUDIT.md`(실제 코드 감사). 코드 미수정, 설계 문서.
> 대상 사용자 = 본인 1명(Strong 헤비유저, 중급+). 목표 = Strong을 더 편하게 대체.
>
> **두 가지 원칙(전 구간 공통):**
> 1. **사용자 입력 최소화** — 묻지 않아도 되는 건 묻지 않는다. 매 세션 반복 입력 금지.
> 2. **자동 계산 최대화** — 직전/최고 기록, 오늘 목표, 증량 신호, 부위별 세트는 전부 앱이 계산해 보여주고, 사용자는 "무게·횟수·완료"와 "마지막 세트 노력도"만 손댄다.
>
> 표기 규칙: 각 입력에 `[현재 받음]`(이미 구현) / `[신규]`(새로 받아야 함) / `[제거 후보]`를 APP_AUDIT 근거로 단다.

## 0. 입력 총량 요약 (한눈에)

| 구간 | 사용자 입력 | 빈도 | 비고 |
|---|---|---|---|
| 1. 온보딩 | 4개(목적·주간횟수·증량트리거·상하체 증량폭) | **평생 1회** | 부위/경력은 묻지 않음(자동 도출) |
| 2. 운동 시작 전 | 0~1개(시작 방식 선택만) | 매 세션 | 추천/빈운동/불러오기 중 탭 1번 |
| 3. 종목 추가 | 1~2개(종목 선택, 역할은 자동·수정만) | 종목 추가 시 | 역할 기본값 자동 |
| 4. 세트 입력 | 2~3개(무게·횟수·완료) | 세트마다 | 핵심 입력. 프리필로 타수 절감 |
| 5. 노력도 | 1개(버튼 4개 중 1) | 마지막 작업세트 후 | 숫자 강요 금지 |
| 6. 운동 종료 | 0개(원칙), 선택형 컨디션 1개(옵션) | 매 세션 | 추가 입력 0 목표 |
| **합계(정상 세션)** | **세트 입력 + 노력도뿐** | | 온보딩 외 거의 무입력 |

## 1. 최초 온보딩 — 평생 1회

목적: 엔진(`ProgressionEngine`/`OverloadService`)이 단계·rep range·증량폭·주간 목표를 계산하는 데 **반드시 필요한 최소값**만 받는다.
근거: `GoalSetting.java` 필드 = `goalType / weeklyFrequency / incUpper / incLower / progressionTrigger / onboardedAt`(APP_AUDIT §10).

### 사용자가 입력하는 값 (4개)

| # | 입력 | UI | 기본값 | 왜 필요한가(이게 없으면) | 상태 |
|---|---|---|---|---|---|
| 1 | **목적** hypertrophy/strength/fatloss/endurance | 4지선다 | hypertrophy | rep range·증량 전략의 분기점. 근력=바벨 3~6, 비대=6~12 등 전부 여기서 갈림(`repRange`) | `[현재 받음]` |
| 2 | **주간 운동 횟수** | 숫자(2~6) | 4 | 출석 목표 도트바·todayPlan·부위별 목표세트 분배의 분모(`weeklySummary` 출석/부위갭) | `[현재 받음]` |
| 3 | **증량 트리거** single / two_sessions / rpe | 3지선다 | two_sessions | "언제 무게를 올릴까" 판정 기준. 엔진에 실제 연결됨(`readyToIncrease(...trigger)`) | `[현재 받음]` |
| 4 | **상체/하체 증량폭(kg)** incUpper/incLower | 숫자 2개 | 상2.5 / 하5 | 증량 제안 무게의 기본 단위(`effectiveIncrement`가 여기서 출발, 머신/고립은 절반) | `[현재 받음]` |

### 앱이 자동으로 도출하는 값 (묻지 않음)

| 항목 | 도출 방식 | 왜 안 묻는가 |
|---|---|---|
| **목표 부위** | 종목을 추가하면 `muscleGroup`에서 부위별 주간 세트가 자동 집계됨. 별도로 "가슴 키우고싶다" 안 물어도 종목 선택이 곧 의사표시 | 입력 1개 절감. 부위 선택 화면(`body-parts`)은 있으나 온보딩 필수 아님 |
| **경력(초·중·고급)** | 받지 않음. 기록이 쌓이면 단계 머신이 `NEED_BASELINE→BUILD_REPS→READY_TO_INCREASE`로 자동 분류 | 타겟이 중급+ 본인 1명. 자가신고 경력은 엔진이 안 씀(APP_AUDIT 어디에도 경력 입력 없음) → 물으면 죽은 입력 |
| **종목별 분류**(barbell/machine/isolation/bodyweight) | `equipmentType + muscleGroup`에서 `classifyRuleType`로 파생 | 엔티티에 분류 컬럼 없음(§9). 사용자가 고를 일 아님 |
| **rep range** | 목적+분류로 `repRange`가 단일 소스로 계산 | 사용자가 6~12를 직접 칠 이유 없음 |

> ⚠ 온보딩 게이트: `goalSetting.onboarded`가 false면 홈이 `/onboarding`으로 강제 이동(`index.tsx:47-53`). 즉 위 4개는 **최초 1회 필수**, 이후 설정에서 수정만.

## 2. 운동 시작 전 — 탭 1번(또는 0번)

목적: "오늘 뭐 하지"를 앱이 제안하고, 사용자는 **시작 방식만 고른다**. 새 입력 거의 없음.

### 사용자가 입력하는 값 (0~1개)

| 입력 | 설명 | 상태 |
|---|---|---|
| **시작 방식 선택**(탭 1번) | ① 추천 루틴(`suggestedTemplates`) ② 빈 운동 ③ 최근 세션 불러오기 ④ 템플릿. 넷 중 하나 탭 | `[현재 받음]`(템플릿 시작=마지막 세션 복사, §11) |
| ~~오늘 집중 부위 직접 선택~~ | **묻지 않음**(아래 자동 계산으로 대체) | `[제거 후보]` |

### 앱이 자동 계산해 미리 보여주는 값

| 항목 | 출처 | 사용자 관점 |
|---|---|---|
| **오늘 운동하면** 카드 | `summary.todayPlan` | "오늘 가슴/등 하면 좋아요" 자동 제시 → 부위를 손으로 안 골라도 됨 |
| **목표 부위 주간 세트 부족분** | `summary.bodyPartVolumes`(목표 충족 시 초록) | 어느 부위가 모자란지 색으로 보여줌 → 그게 곧 오늘 집중 가이드 |
| **출석 진행** | `att.done/target` 도트바 | 이번 주 몇 번 했는지 |
| **추천 루틴** | `suggestedTemplates` | 빈 화면에서 시작 안 해도 됨 |

> 설계 의도: "오늘 집중 부위"를 **입력으로 받지 말고 자동 제안으로 대체**. 사용자가 부족 부위를 보고 추천 루틴을 탭하면 끝(입력 1번 → 0~1번).

## 3. 종목 추가 시 — 종목 선택 1번, 역할은 자동

목적: 종목을 고르면 역할(핵심/보조/기록만)과 추적여부는 **앱이 기본값을 정하고**, 사용자는 틀렸을 때만 고친다.

### 사용자가 입력하는 값 (1~2개)

| 입력 | UI | 기본값/자동 | 상태 |
|---|---|---|---|
| **종목 선택** | 검색·리스트 탭 | — (필수 1번) | `[현재 받음]`(workout.tsx 종목선택) |
| **역할** core / support / log_only | 배지 탭(수정 시에만) | `classifyRole`이 자동 배정. **기본은 자동**, 사용자는 덮어쓰기만 | `[현재 받음]`(ExerciseGoalSheet 역할 수정, `ExerciseGoal.role/overridden`) |
| **목표 추적 여부** | 역할에 종속 | `log_only` 선택 = 추적 끔. 별도 토글 불필요 | `[현재 받음]`(log_only는 평가 제외, §11) |

### 앱이 자동 계산하는 값

| 항목 | 출처 |
|---|---|
| 종목 분류(barbell/machine/isolation/bodyweight) | `classifyRuleType` |
| rep range·목표 세트수 | `repRange` / `targetSets` |
| 증량폭 | `effectiveIncrement`(머신·고립 절반) |
| 현재 단계 배지 | `compute → stageLabel` |

> 설계 의도: "역할"을 **추가할 때 강제로 묻지 말 것**. 자동 배정 후, 종목 카드/시트에서 사용자가 바꿀 수 있게(현재 `overridden` 플래그가 이 의도와 일치). 목표추적 on/off를 별 토글로 만들지 말고 `log_only` 역할로 흡수 → 입력 1개 절감.

## 4. 세트 입력 중 — 핵심 입력(무게·횟수·완료)

목적: 헬스장에서 실제로 손대는 유일한 반복 입력. **3초 룰** 충족이 최우선. 프리필로 타수를 줄인다.

### 사용자가 입력하는 값 (세트당 2~3개)

| 입력 | UI | 절감 장치 | 상태 |
|---|---|---|---|
| **무게(kg)** | NumPad / ± 버튼 | 직전 세트값 프리필, ± 조정(`handleAdjustWeight`) | `[현재 받음]` |
| **반복수** | NumPad | 직전 세트값 프리필 | `[현재 받음]` |
| **완료** | 체크 탭 | 1탭. 저장 실패 시 완료 안 됨(desync 방지) | `[현재 받음]`(`handleCompleteSet`) |
| (선택) **세트 타입** NORMAL/WARMUP/DROP/FAILURE | 순환 탭 / 롱프레스 | 기본 NORMAL, 안 건드려도 됨 | `[현재 받음]`(`handleCycleSetType`) |
| 시간기반 종목 | 초(sec) 입력 | trackingType=TIME일 때만 | `[현재 받음]` |

### 운동 중 화면에 **자동으로 떠야 하는** 값 (현재 미흡 → 신규)

| 항목 | 출처 | 현재 상태 | 사용자 관점 |
|---|---|---|---|
| **오늘 목표(todayTarget)** | `Guide.todayTarget` | `[신규]` 운동 중 화면에 약하게 노출(§13-2) | "오늘 80kg 8회" 가 세트 옆에 떠야 3초 룰 충족 |
| **남은 반복수(targetTotalReps)** | `ExerciseGoalDto.targetTotalReps` | `[신규]` workout.tsx 활용 흔적 약함 | "총 24회 중 9회 남음" |
| **증량 신호(READY_TO_INCREASE 배지)** | `stage` | `[신규]` 운동 중 미노출 | "오늘 올릴 때" 배지 |
| **다음 무게 프리필** | `computeNextTarget`/`effectiveIncrement` | `[신규]` 자동 프리필 없음(§13-3) | 증량 준비 종목은 다음 무게 자동 채움 → 사용자는 수락만 |
| 직전 기록 | `formatRecord` | `[현재 받음]`(카드에 지난기록) | "지난번 77.5kg 8회" |
| 1RM·PR 표시 | Epley·역대최고 비교 | `[현재 받음]` 완료 시 PR 햅틱 | |

> ⚠ 설계 의도(P0·P1): **운동 중 화면에 오늘목표/증량신호/다음무게 프리필을 노출**하면 사용자 입력이 줄어든다(무게를 직접 안 쳐도 됨). 현재는 홈/시트/상세에만 있고 정작 운동 중엔 약함 → 입력 최소화의 가장 큰 미수확 지점.

## 5. 마지막 세트 후 노력도 — 버튼 4개(숫자 강요 금지)

목적: 엔진의 "여유" 판정·`rpe` 트리거에 들어갈 노력도를 **숫자 없이** 받는다. 작업세트마다가 아니라 **마지막 작업 세트 1회만**.

### 사용자가 입력하는 값 (1개, 버튼 택1)

| 버튼 | 의미(RIR 매핑) | 엔진 해석 |
|---|---|---|
| **여유 많음** | RIR 4+ | 증량 신호 강함 |
| **2~3개 남음** | RIR 2~3 | 정상 진행 |
| **거의 한계** | RIR 1 | 유지 권장 |
| **실패** | RIR 0 | 증량 보류/디로드 신호 |

- UI: 마지막 작업 세트 완료 직후 바텀시트 "마지막 세트 어땠나요?"(`RIR_OPTIONS`, `workout.tsx:2227-2243`).
- **숫자(1~10 RPE) 직접 입력 금지** — 헬스장에서 부담. 버튼 4개로 충분.

### 상태 — ⚠ 치명 격차(P0)

| 항목 | 현재 | 필요 |
|---|---|---|
| RIR 팝업 UI | `[현재 받음]` 존재 | — |
| 저장 | `[신규]` `rirSession`은 **쓰기만 하고 안 읽음** → 즉시 폐기(§4·§13-1) | 세트 모델에 `rir/effort` 컬럼 추가, 세트 저장 시 동봉 |
| 엔진 연결 | `[신규]` `evaluate`/`readyToIncrease`가 RIR 미사용, `rpe` 트리거가 `single`로 폴백(§11-1) | 노력도를 실제 판정에 투입 |

> 즉, **현재 노력도 입력은 받기만 하고 버려진다.** 이 입력이 살아나야 5번 구간이 의미를 가진다(모델→전송→엔진 3단 연결 필요). 그 전까지는 "여유 많음" 성공조건이 데이터 없는 안내 문구일 뿐.

## 6. 운동 종료 후 — 추가 입력 0 (원칙)

목적: 운동을 끝내면 **아무것도 더 묻지 않는다.** 요약은 앱이 자동 생성. 컨디션은 정말 필요할 때만 선택형 1개.

### 사용자가 입력하는 값 (0개 권장)

| 입력 | 권장 | 상태 |
|---|---|---|
| **종료(완료) 탭** | 1탭(입력 아님) | `[현재 받음]`(`handleFinishWorkout`) |
| (옵션) **컨디션** 좋음/보통/피곤 | **선택형 1개만**, 스킵 가능. 숫자·텍스트 금지 | `[신규]` 현재 없음. 추가 시에도 필수 아님 |
| ~~메모·제목·태그 강제~~ | 종료 시 강제로 묻지 말 것. 인라인 편집은 운동 중 선택적으로만 | `[현재 받음]`(운동 중 인라인, 종료 게이트 아님) |

### 앱이 자동 생성하는 종료 요약 (입력 0)

| 항목 | 출처 |
|---|---|
| 총 볼륨·세트수·종목수·PR | `proceedFinish` 요약 계산 |
| 세션 평가(개선/유지/보류) | `getSessionEval(sessionId)` → `evaluate`(IMPROVED/MAINTAINED/MISSED/...) |
| 다음 브리핑 dirty 플래그 | `completeSession` |

> 설계 의도: 종료 화면은 **읽기 전용 요약**이 기본. 컨디션을 넣더라도 "선택형 1개·스킵 가능"을 넘기지 말 것(메모리: 사용자 입력 최소화).

## 7. 앱이 자동 계산하는 값 — 전체 목록

사용자가 절대 손으로 입력하지 않고, 엔진/통계가 계산해 보여주는 값. (출처 = APP_AUDIT §11·§12)

### 비교 기록

| 값 | 출처 | 표시 위치 |
|---|---|---|
| **직전 기록** | `formatRecord` / `bestRecordOf` 직전 | 종목 카드·시트·운동 중 |
| **최고 기록(이 종목)** | `bestRecordOf` | 시트 비교기준 |
| **전체 PR(1RM)** | `allTimePrOf` / Epley 역대최고 | 시트, 종목 상세 헤더 |
| **1RM 추이** | `dailyE1rm` / `get1RMHistory` | 종목 상세 차트 |
| **렙별 기록(1~12RM)** | `getRepMaxes` / `RepTable`(Epley) | 종목 상세 |

### 오늘/다음 목표

| 값 | 출처 |
|---|---|

## 부록: 입력 최소화 미수확 지점 (설계 우선순위)

APP_AUDIT §15와 정합. "입력을 더 줄이거나 살리는" 관점 정렬:

1. **[P0] 노력도(5번) 살리기** — 지금은 받고 버림. 저장+엔진 연결돼야 그 1개 입력이 가치를 가짐. 안 그러면 차라리 안 묻는 게 정직.
2. **[P1] 운동 중 다음 무게 자동 프리필(4번)** — 증량 준비 종목은 무게를 손으로 안 치게. 입력 1개/세트 절감.
3. **[P1] 운동 중 오늘목표·증량배지 노출(4번)** — 입력은 안 줄지만 3초 룰 충족(지난기록 보러 화면 이동 불필요).
4. **[유지] 온보딩 4개·부위/경력 안 묻기(1번)** — 현 설계가 이미 최소. 더 묻지 말 것.
5. **[유지] 종료 시 추가 입력 0(6번)** — 컨디션 넣더라도 선택형 1개·스킵 가능 상한 지킬 것.

### 핵심 6줄
1. 온보딩은 **평생 4개**(목적·주간횟수·증량트리거·상하체 증량폭)만 받고, 부위·경력·분류·rep range는 전부 자동 도출 — 더 묻지 않는다.
2. 정상 세션에서 사용자가 손대는 건 사실상 **세트의 무게·횟수·완료**와 **마지막 세트 노력도 버튼 1개**뿐, 나머지(시작 추천·역할·요약)는 앱이 계산·제안한다.
3. 노력도는 숫자 금지·**버튼 4개**(여유 많음/2~3개/거의 한계/실패)로 받되, **현재는 받고 즉시 폐기**(`rirSession` 미사용)되어 P0로 모델·전송·엔진 연결이 필요하다.
4. 운동 중 화면에 **오늘 목표·증량 배지·다음 무게 프리필**이 약해(P1) 입력 최소화/3초 룰의 최대 미수확 지점 — 증량 준비 종목은 무게를 자동 채워 "수락만" 하게 해야 한다.
5. 운동 종료 후 추가 입력은 **0이 원칙**, 컨디션을 넣더라도 선택형 1개·스킵 가능 상한을 넘기지 않는다.
6. 직전/최고/전체 PR·오늘목표·성공조건·다음목표·비교신뢰도·정체가능성·부위별 주간세트·부족분은 **전부 엔진/통계 자동 계산** — 사용자 입력 항목이 아니다.
