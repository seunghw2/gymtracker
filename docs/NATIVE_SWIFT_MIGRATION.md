## 개요

지금 결정된 것은 "기존 GymTracker에서 프론트를 새로 만든다"이고, 백엔드(카카오·애플 로그인·JWT·CRUD·통계·AI 리포트·점진과부하 엔진 자리)는 유지한다는 것이다. 원래 계획은 Expo를 유지한 채 검증된 배관(카카오·API 클라이언트·유틸)만 새 레포로 이식하는 것이었으나, 사용자가 **애플워치 + 워치로 측정한 수면·회복 데이터(HealthKit) 활용**을 이유로 SwiftUI 네이티브 전환을 진지하게 검토하기를 요청했다.

이 문서는 그 전환을 현실적으로 분석한다. 결론을 먼저 요약하면 — **전환의 유일한 실질 이득은 애플워치·HealthKit 네이티브 접근이고, 그마저도 "본체는 그대로 두고 워치·헬스킷만 얇게 붙이는" 하이브리드로 대부분 얻을 수 있다.** 반면 비용(프론트 전량 재작성, 검증 파이프라인 상실, 리모트 개발환경과의 마찰)은 크고 확정적이다. 그래서 이 결정의 진짜 기준은 "Swift가 더 좋은가"가 아니라 **"애플워치·수면을 제품의 중심 차별화로 삼을 것인가"** 하나다.

### 사용자 개발환경 제약 (반드시 감안)

- 사용자는 **아이폰으로 리모트 Mac에 접속**해 개발한다. 터미널·GUI를 직접 만지는 데 제약이 잦고, 명령은 Claude가 대행한다.
- Expo는 이 환경에 잘 맞았다 — `expo start --tunnel`을 띄우면 폰 Expo Go로 즉시 로드되고, 코드 저장 시 핫리로드된다.
- Claude는 프론트 변경 후 **Expo Web + Playwright**로 헤드리스 화면을 직접 띄워(토큰 주입) 렌더 깨짐을 잡는 무인 검증을 해 왔다.
- Swift는 매 변경마다 Xcode(또는 xcodebuild) 빌드가 필수고, 실기기 구동은 프로비저닝·서명이 걸린다. `simctl`(시뮬레이터 CLI)은 이 자동화 셸에서 hang한 이력이 있다.
- 이식하려던 배관(카카오 로그인·API 클라이언트·유틸)은 전부 TypeScript라 Swift로 가면 재사용이 안 된다. 단, 백엔드 REST 연동은 언어와 무관하다.

## 1. 전환 타당성 — 득과 실

전환을 저울에 올릴 때 득과 실이 서로 다른 층위에 있다는 점을 먼저 봐야 한다. **득은 대부분 "제품 기능"의 영역**(워치·수면·햅틱·라이브 액티비티)이고, **실은 대부분 "개발 워크플로우"의 영역**(빌드·서명·검증 상실)이다. 그래서 "Swift가 기술적으로 우월한가"를 묻는 건 초점이 어긋난다 — 우월성보다 이 두 층위의 교환을 감당할 가치가 있는지가 질문이다.

### 1-1. 네이티브의 실익

**애플워치·HealthKit 네이티브 접근(가장 크고, 사실상 유일하게 정당한 이유).** 워치 앱으로 손목에서 세트 확정·햅틱 휴식타이머·심박존을 다루고, HealthKit로 수면·HRV·안정시심박·활동에너지를 읽어 회복 상태를 코칭에 반영한다. 이건 2장에서 따로 깊게 다룬다.

**게임형 연출·햅틱이 확실히 좋아진다.** 확정 화면의 핵심인 2층 게이지(막대=세션 채움 + 흰 게이트 마커=추세 해금)와 게이트 점등 연출은 SwiftUI에서 `Canvas`·`TimelineView`·`phaseAnimator`(iOS 17)로 네이티브 렌더된다. 게이트 통과·신기록 순간(결정 4의 하이브리드 축하 1장)의 커스텀 햅틱은 Core Haptics로 밀리초 단위 진동 패턴(rise→pop)을 설계할 수 있다. Expo/RN은 `expo-haptics`의 프리셋 몇 개로 제한된다. "핵심 순간 연출"이 제품 정체성이라면 이 차이는 체감된다.

**백엔드에 애플 로그인이 이미 완비돼 있다.** 백엔드에 Sign in with Apple(identityToken 검증 + JWKS) 엔드포인트가 이미 구현돼 있다. iOS 네이티브면 `AuthenticationServices`의 애플 로그인 버튼으로 원탭 로그인 → identityToken을 기존 엔드포인트에 그대로 던지면 끝이라, 카카오 SDK 재구현 부담이 거의 사라진다(타겟이 본인 1명이면 애플 로그인만으로 충분).

**서드파티 의존성이 준다.** HealthKit·Core Haptics·SwiftData·AuthenticationServices·Charts가 전부 1st-party라, RN처럼 `expo-*`/`react-native-*` 십수 개를 얹고 버전 충돌을 관리할 필요가 없다. 유지보수 표면이 줄어드는 건 실제 이점이다.

**아이폰 전용이라 크로스플랫폼 세금이 없다.** 타겟이 본인(아이폰 유저)이라 안드로이드 포기의 손실이 거의 없다. 다만 이건 Swift의 근거가 아니라 별개 결정이다 — Expo도 iOS만 빌드하면 그만이고, "아이폰 전용"과 "Swift 전환"은 분리해서 봐야 한다.

### 1-2. 비용과 리스크 (정직하게)

**프론트 18.6K줄이 전량 폐기된다.** 현 프론트는 TS/RN으로 화면 31·컴포넌트 32·lib 41개 규모다. SwiftUI로 가면 운동 중 화면(듀얼 게이지·게이트 자물쇠·세트 카드)·데일리 랩업·로컬 저장·탭 네비·상태관리·차트·드래그 리스트·사운드/햅틱/알림 배선이 전부 처음부터 다시 짜인다. RN 코드는 한 줄도 안 넘어간다.

**검증된 TS 배관이 재사용 불가다.** 카카오·구글 로그인 연동, API 클라이언트(401→refresh→retry), date/format/units 유틸(약 1.5K줄), 토큰 저장은 전부 TypeScript다. Swift면 카카오는 Kakao iOS SDK로, 애플 로그인은 ASAuthorization으로, 나머지는 전부 새로 짠다. (RN을 유지하면 새 레포라도 이 배관은 복붙 수준으로 산다.)

**개발 워크플로우가 깨진다 — 이게 실질 최대 비용이다.** 사용자는 아이폰으로 리모트 Mac을 쓰고 쉘을 직접 못 쳐서 Claude가 대행한다. Expo Go의 "폰에서 즉시 리로드"가 사라지고 매 변경마다 빌드가 필요하다. 실기기 구동의 첫 서명 설정은 사실상 Xcode GUI 상호작용이 필요한데, 리모트+아이폰 화면으로 이걸 잡는 건 고통스럽고 Claude가 GUI를 대신 못 만진다. 빌드 실패 로그도 아이폰 화면으로 읽기 나쁘다.

**Claude의 무인 화면검증 루프가 끊긴다.** 지금은 프론트 변경 후 Claude가 Expo Web + Playwright로 헤드리스 화면을 직접 띄워 렌더 깨짐을 잡는다(리팩터링 안전망). SwiftUI엔 이 무인 경로가 없다 — `#Preview`는 Xcode GUI 안에서만 살고, `simctl`은 이 환경에서 hang한 이력이 있다. 결국 Swift는 "사용자가 폰으로 직접 확인"에 전적으로 의존하게 되고, Claude가 짜고 사용자가 확인하는 왕복이 느려진다.

**Swift/SwiftUI 첫 학습곡선.** 상태관리(`@State`/`@Observable`), 네비게이션(`NavigationStack`), 비동기(async/await), HealthKit 권한·백그라운드 전달, 워치 커넥티비티(WCSession) 페어링은 각각 함정이 있어 초기 몇 주는 확실히 느려진다. 사용자와 Claude 모두 RN 컨텍스트가 쌓여 있다는 점도 반대 방향의 관성이다.

**애플 개발자 계정·서명은 어차피 든다(전환의 근거는 아님).** 워치·HealthKit·라이브 액티비티·실기기 설치는 언어와 무관하게 Apple Developer Program(연 99달러)과 서명이 필요하다. Expo로 워치를 붙여도 EAS build·서명이 똑같이 발생한다. 즉 "워치를 쓸 거면 어차피 유료 계정·dev build"는 양쪽 공통 비용이라 Swift 전환의 추가 근거가 못 된다. 뒤집으면, **워치를 쓰기로 한 순간 Expo Go 즉시 테스트의 이점은 어차피 일부 포기**라는 뜻이기도 하다.

### 1-3. 살아남는 것 — 백엔드는 언어 무관

이 전환을 "덜 무섭게" 만드는 유일한 요소는 **앱의 진짜 지식자산이 백엔드에 있다**는 점이다. 점진과부하 엔진(4-Strategy·게이트 추세·랩업 묶음)·인증·통계·AI 리포트는 REST/JSON이라 클라이언트 언어와 무관하다. Swift든 RN이든 같은 API를 호출한다. 즉 화면·배관은 재작성 대상이지만, "무엇을 얼마나 올릴지"를 계산하는 두뇌는 전환 리스크 밖에 있다. (자세한 건 4장.)

## 2. 애플워치 + 수면·회복 데이터 활용

여기가 사용자가 Swift에 끌린 핵심이자, 전환을 정당화할 수 있는 유일한 기능 영역이다. 그래서 가장 깊게 본다. 다만 결론부터 정직하게 말하면 — **회복 데이터는 판정을 새로 만드는 1차 신호가 아니라, 이미 있는 신호를 강화·중립화하는 보조축**으로만 써야 한다. 그렇게 써야 확정 스펙과 충돌하지 않고, 데이터가 부정확할 때 사용자를 잘못 벌주지 않는다.

### 2-1. 왜 지금 구조로는 불가능한가

현재 앱은 Expo Go로 폰 즉시 테스트하는 구조다. HealthKit·WatchConnectivity·WidgetKit·라이브 액티비티(ActivityKit)는 전부 커스텀 네이티브 모듈이라 **Expo Go 런타임에는 절대 안 들어온다.** 즉 "수면 읽어서 코칭"을 하려면 최소한 EAS Dev Build(config plugin + custom dev client)가 필요하고, 워치 앱까지 가려면 Xcode 워치 타깃이 필수다. **핵심: HealthKit을 진지하게 쓰기로 한 순간, 언어와 무관하게 Expo Go 워크플로는 포기된다.** 그러면 RN(EAS Build)이든 SwiftUI든 "빌드·서명 필요"는 같아지고, 차이는 "RN은 TS 배관 재사용 가능 vs SwiftUI는 전면 재작성"뿐이다.

### 2-2. 무엇을 읽고, 정확도는 어떤가

HealthKit로 읽을 대상과 각각의 현실적 정확도:

- **수면(sleepAnalysis).** iOS16+·watchOS9+부터 Core/Deep/REM 단계까지 구분된다. **단, 워치를 차고 자야만** 단계가 나온다. 폰만 있으면 "침대에 있던 시간(inBed)"뿐이라 회복 신호로 약하다. 헤비 유저도 매일 워치를 차고 자지는 않는다.
- **HRV(SDNN).** 워치가 밤중·안정시 산발 측정. 개인 내 변동이 커서 **절대값이 아니라 "본인 7일 베이스라인 대비 편차"로만** 의미가 있다. 베이스라인이 없는 첫 1~2주는 판정을 못 한다.
- **안정시심박.** 상승은 과훈련·수면부족·질병의 지연 신호로 비교적 견고하다.
- **활동에너지·걸음·운동시간.** 보조.

원칙은 하나다 — **단일 지표 절대값으로 판정하지 말고 "본인 rolling baseline 대비 오늘의 편차"로만 쓰고, 데이터가 없으면 조용히 끈다.**

### 2-3. 확정 스펙의 어디에 꽂히는가 (정확한 자리)

회복 데이터는 새 기능을 만드는 게 아니라 **이미 확정된 로직의 "빈칸"을 메운다.** 그 빈칸이 구체적으로 세 곳이다.

**(a) 결정 6 — "동시 정체 → 디로드 신호"의 근거줄.** 확정 결정 6은 탑·백오프가 둘 다 2~3주 정체여도 자동 디로드를 처방하지 않고 "디로드 고려" 신호만 띄운다. 그 이유가 문서에 명시적으로 "디로드는 수면·생활 스트레스 등 **앱이 모르는 변수**에 좌우돼 거짓 양성 위험이 크다"이다. HealthKit이 바로 그 "앱이 모르는 변수"를 부분적으로 아는 앱으로 바꾼다. 적용은 "디로드 고려" 카드의 근거 리스트에 한 줄 추가하는 형태다 — 예: "최근 5일 안정시심박 +6bpm · HRV 본인평균 −18% · 수면 6.2h(평소 7.4h) → 회복 부족이 정체에 겹칩니다." 반대로 회복이 멀쩡하면 "회복 지표는 정상 → 정체는 프로그램 이슈일 수 있어요(세트+1·보조종목 먼저)"로 신호의 방향을 가른다. **중요: 이건 여전히 신호 강화지 자동 처방이 아니다.** 결정 6의 "자동 아님" 배지·사용자 선택 원칙은 그대로 유지되고, 회복 데이터가 하는 일은 "이 신호를 얼마나 믿을지"의 confidence를 사용자에게 정직히 보여주는 것이다.

**(b) 케이스 매트릭스 — "컨디션 난조 일괄 할인" 칩의 프리필.** 케이스 매트릭스에 이미 "컨디션 난조 전체 하향 → 첫 종목 큰 하락 시 좋음/평범/난조 1탭 칩 → 난조면 전 종목 후퇴를 일괄 단발 할인"이 있다. 지금은 이 판단이 100% 사용자 1탭에 의존한다. HealthKit이 이 칩을 프리필한다 — 세션 시작 시 "어젯밤 수면 5.9h·안정시심박↑"가 감지되면 컨디션 칩을 "난조?" 쪽으로 기본 하이라이트한다(강제 아님, 사용자가 뒤집을 수 있음). 그러면 헬스장에서 컨디션 자가입력을 깜빡해도 그날 나온 하락이 잘못 REGRESSED로 추세를 오염시키는 걸 막는다. **판정 로직은 그대로, 입력 부담만 0에 더 가까워진다.**

**(c) 세션 전 "오늘 톤" 배너.** 회복이 나쁜 날 세션 시작 시 "오늘은 유지/가볍게 권장" 배너로 세션 난이도를 사전 하향한다. 이건 개별 세트 RIR을 바꾸는 게 아니라 세션 전체의 목표 톤을 낮추는 것이다.

### 2-4. RIR 대체·보완의 한계 (과장 금지)

사용자 관심 중 하나가 "수면·HRV로 RIR을 대체·보완"인데, 여기가 가장 정직해야 할 지점이다. **RIR(남은 반복 여유)은 세트 단위의 국소 강도 신호이고, HRV·수면은 하루 단위의 전신 회복 신호다.** 해상도와 대상이 다르다. HRV가 낮은 날 그 세트가 몇 개 남았는지는 예측 못 한다. 따라서 "수면으로 RIR을 대체"는 과장이다. 현실적 역할은 둘로 한정된다:

1. **세션 톤 조절** — 회복 나쁜 날 세션 난이도 사전 하향(개별 세트 RIR이 아니라).
2. **RIR 추세 해석의 컨텍스트** — 확정 로직은 "쉬워짐(증량 가속)"을 단일 세션이 아니라 같은 무게에서의 RIR 추세 하락으로만 판정한다. 여기서 회복 나쁜 날의 EASY를 신뢰하지 않도록 할인해 "거짓 EASY"를 방지한다.

즉 회복 데이터는 RIR을 **대신하는 게 아니라 RIR 해석의 컨텍스트**다. 확정 로직의 "쉬워짐은 RIR 추세로만"과 결이 같다.

### 2-5. 애플워치 "앱" — 손목 로깅·햅틱 휴식타이머

HealthKit "읽기"와 별개로, 워치에 독립 앱(watchOS 타깃)을 올리면 헬스장 실사용 가치가 크다:

- **손목 세트 확정** — 무게·반복은 폰에서 계획해두고, 워치에선 Digital Crown으로 반복만 조정 + 크라운 클릭으로 세트 완료. (전체 입력을 워치에서 하긴 화면이 좁아 비현실적. "확정"만 손목이 현실적.)
- **햅틱 휴식타이머** — 세트 완료 시 자동 시작, 종료 3초 전·종료 시 손목 탭. 폰 안 꺼내고 손목 진동으로 아는 건 확정 스펙의 "헬스장 3초 룰"과 매우 잘 맞는다.
- **세트 간 심박 회복(HR recovery)** — 세트 직후 심박이 얼마나 빨리 떨어지는지를 회색 참고값으로 병기(판정엔 안 씀).
- **HKWorkoutSession** — 운동을 워치 워크아웃 세션으로 열면 심박·칼로리가 정확히 기록되고 애플 활동링·피트니스에 남아 "운동으로 인정"된다(연속기록 동기).

**비용의 현실:** 워치 앱은 별도 UI·별도 상태동기(WatchConnectivity)·별도 QA가 필요한 사실상 "두 번째 앱"이다. MVP에 넣으면 범위가 1.5배로 늘고, Claude가 Playwright로 자동 검증할 수 없는 영역이라 회귀 위험이 크다. 게다가 타겟이 워치 항시착용자가 아니다.

### 2-6. 라이브 액티비티·다이나믹 아일랜드·위젯

**라이브 액티비티(ActivityKit)** — 휴식 타이머를 잠금화면·다이나믹 아일랜드에 카운트다운으로 띄우면, 앱을 안 열어도 폰이 잠겨 있어도 남은 휴식이 보인다. 헬스장에서 세트 사이 폰을 잠깐 볼 때 최고의 순간이고, 네이티브가 압도적으로 유리하다(Expo Go 완전 불가, RN은 EAS+config plugin으로 가능하나 손이 많이 감, SwiftUI는 1급 지원). **위젯(WidgetKit)** — "오늘의 퀘스트" 3종·이번 주 부위 볼륨 게이지를 홈 위젯으로 노출하면 앱 열기 전부터 동기가 생긴다. 우선순위는 라이브 액티비티 휴식타이머 > 위젯 > 워치 앱 순이다. 라이브 액티비티 하나만으로도 "네이티브 감성"이 확 산다.

### 2-7. 권한·프라이버시·데이터 결손

- **HealthKit 권한은 타입별로 개별 허용/거부되고, 앱은 "거부됐는지조차" 알 수 없다**(프라이버시 정책상 read 거부 = "데이터 없음"과 구분 불가). 그래서 코드는 항상 "데이터가 안 온다"를 정상 경로로 다뤄야 하고, 회복 코칭은 무조건 opt-in·있으면 켜고 없으면 조용히 꺼야 한다.
- **온보딩은 맥락적으로** — 첫 실행 강제 대신 "정체 신호가 처음 뜰 때" 또는 설정에서 "회복 기반 코칭 켜기"로 요청하는 게 허용률이 높다.
- **데이터는 온디바이스 처리 원칙** — 수면·HRV 원자료를 백엔드로 보내면 민감정보 취급·심사 리스크가 커진다. 회복 인사이트 계산은 폰에서 하고, 백엔드엔 "난조 플래그" 같은 파생 요약값만 보낸다(4장 참조).
- **워치 없는 날 폴백(절대 원칙)** — 타겟은 아이폰 헤비유저지 워치 항시착용자가 아니다. 회복 기능은 "있으면 보너스, 없으면 완전 정상 동작"이어야 한다. 오늘 데이터 없으면 컨디션 칩 프리필 안 함, 베이스라인 부족(첫 2주)이면 회복 신호 전면 보류. **워치 데이터 없다고 사용자를 불이익 주거나 "워치 사세요" 넛지하는 건 절대 금지.** 이건 확정 스펙의 케이스 매트릭스가 이미 세운 "데이터 부족 → INSUFFICIENT → 보류" 철학과 정확히 같은 규율이다.

### 2-8. 게임형 동기부여와의 결합

디자인 방향(듀얼 게이지·데일리 랩업·핵심순간 연출)과 애플 생태계가 만나는 지점:

- **데일리 랩업에 "회복 컨텍스트" 한 줄** — "어젯밤 잘 자고(7.8h) 게이트 2개 통과 — 최상의 날" 또는 "수면 부족에도 유지 — 잘 버텼어요"로 그날 서사에 회복을 엮으면 숫자가 스토리가 된다.
- **활동링 이중 보상** — HKWorkoutSession으로 운동이 애플 활동링을 채우면 "앱 밖에서도 인정받는" 보상.
- **위젯의 상시 노출** — "오늘의 퀘스트" 게이지가 홈화면에 상시 보인다.

**단 경고:** 회복 데이터를 게임 점수화(수면점수로 랭크 매기기 등)하면 데이터가 부정확할 때 사용자를 잘못 벌준다. 회복은 "점수"가 아니라 "오늘의 컨텍스트/서사"로만 써야 이 앱의 진지·정확 원칙에 맞는다.

### 2-9. MVP 범위 권장 — 단계적으로

- **P0(MVP): 회복 기능 0.** 확정된 진행 엔진·운동 중 화면·데일리 랩업을 먼저 완성한다. 회복 데이터는 정체·디로드 로직이 실제로 굴러가야 얹을 자리가 생긴다 — 순서를 뒤집으면 안 된다. 단, HealthKit "읽기 배관"과 권한 온보딩 문구·베이스라인 저장소는 아키텍처 설계 시점에 미리 자리를 비워둔다.
- **P1(빠른 후속, 가치 대비 비용 최고):** HealthKit 읽기 opt-in — 수면·HRV·안정시심박 베이스라인 → (a)디로드 신호 근거줄, (b)컨디션 난조 칩 프리필, (c)세션 톤 배너. + 라이브 액티비티 휴식타이머(네이티브 감성 즉효).
- **P2(확장):** 애플워치 앱(햅틱 휴식타이머·손목 세트확정·세트간 심박·HKWorkoutSession), 위젯("오늘의 퀘스트").

이유: 워치 앱은 개발·검증 비용이 크고 자동검증이 불가라 코어가 안정된 뒤가 안전하다. 라이브 액티비티와 HealthKit 읽기는 상대적으로 국소적이라 P1이 현실적이다.

## 3. SwiftUI 아키텍처·게임형 UI·인증·데이터

Swift로 간다고 가정할 때의 구현 방향이다.

### 3-1. 앱 구조 — MVVM + 엔진은 별도 모듈

권장 구조:

1. **OverloadEngine 모듈(순수 Swift, UI/네트워크 무의존)** — 만약 엔진 일부를 클라에 두게 되면 여기에. 단, 4장에서 논하듯 엔진 본체는 서버 유지가 기본값이라, 이 모듈은 "오프라인 잠정 게이지 계산" 같은 단순 산술로 최소화한다.
2. **DataStore** — SwiftData `@Model` + Repository.
3. **API 레이어** — URLSession + async/await + 토큰 refresh를 다루는 actor. RN `api.ts`의 401→refresh→retry 흐름만 약 40줄 포팅한다.
4. **Feature별 View + ViewModel** — 운동 중 / 홈 / 랩업 / 분석 / 목표. ViewModel은 `@Observable`(iOS 17)로 zustand를 대체한다.
5. **공유 UI·Effects** — DualGauge·GateMarker·CARBON 컬러 토큰, 그리고 Haptics·Confetti(Canvas) 같은 게임형 연출 유틸.

### 3-2. 게임형 UI 구현

2층 게이지는 `Shape(trim)` 또는 `Canvas`로 막대를 그리고 `.animation(.spring)`으로 채움을, 게이트 마커 점등은 `matchedGeometryEffect`/`phaseAnimator`로 상태 전환을 선언적으로 묶는다. 파티클/글로우는 `TimelineView` + `Canvas`로 네이티브 렌더한다. 핵심 순간 연출은 Core Haptics로 진동+비주얼+사운드를 싱크한다. 데일리 랩업 스토리 장면 전환은 `TabView(.page)`/전체화면 커버 + spring. **화면은 progressionRule을 switch하지 않고 `ProgressionResult`(axes·gate·suggestions)만 받아 그린다** — 확정 심화설계의 원칙을 그대로 옮긴다. 게이지 컴포넌트는 `axisKey`(INTENSITY/VOLUME/TOTAL/EFFORT)만 보고 색·라벨을 매핑한다.

### 3-3. 인증 — 애플 우선, 카카오 선택

애플 로그인은 백엔드에 이미 완비돼 있으므로 `ASAuthorizationAppleIDButton` → identityToken을 기존 `/auth/apple`에 던지는 게 가장 매끄럽다(리뷰 요건상으로도 소셜 로그인이 있으면 애플 로그인 제공이 권장된다). 카카오는 Kakao iOS SDK(SPM)로 병행 가능하나, 타겟이 본인이면 애플 로그인만으로 충분해 auth 재구현 부담이 거의 없다.

### 3-4. 로컬 데이터 — SwiftData 권장

확정 스펙이 요구하는 신규 스키마가 SwiftData와 잘 맞는다 — 세트의 `setType`(NORMAL/WARMUP/DROP/TOP_SET/BACKOFF)·`effort`(EASY~FAILURE/null)·`supersetGroup`·`durationSec`, 목표의 `progressionRule`·`anchorTier`·`effortTracking`·`gateWindow`·`ruleParams`(JSON). 신규 스키마라 SQLite ALTER보다 `@Model`로 처음부터 정의하는 게 유리하고, `ruleParams`는 JSON 수동 파싱 없이 Codable struct 프로퍼티로, enum은 `Codable` Swift enum으로 타입 안전하게 표현된다. derived 값(게이지·게이트)은 저장하지 않고 `@Query` + computed로 재계산한다(확정 스펙의 derived 원칙과 일치).

**SwiftData vs CoreData:** SwiftData는 코드량이 적고 SwiftUI와 직결되지만 iOS 17+ 강제·초창기 마이그레이션 버그 이력이 있다. CoreData는 성숙·세밀 제어가 되지만 보일러플레이트가 많다. 타겟이 본인 최신 아이폰이고 신규 앱이므로 **SwiftData** 권장 — 단 `VersionedSchema`로 마이그레이션 규칙을 앱 초기에 미리 잡아둔다.

### 3-5. 백엔드 연동

네트워킹은 URLSession async/await. 엔드포인트 계약(DTO)은 백엔드가 그대로 제공하므로 언어 무관이다. 로컬-우선 로깅(4장)이면 세트/세션은 로컬에 즉시 쓰고 백그라운드 sync한다.

## 4. 엔진 위치·백엔드 관계·데이터 흐름

이 장이 전환에서 리스크가 가장 낮은 영역이다 — 앱은 **이미 "서버 엔진 + REST" 구조**이기 때문이다.

### 4-1. 현 아키텍처 실측 — 이미 서버 엔진

확인된 사실: 프론트의 세션/세트 CRUD는 전부 라이브 API 호출이고, 프론트에 있는 SQLite 스키마 정의는 앱 코드 어디서도 참조되지 않는 죽은 코드다(즉 지금도 "오프라인 우선"이 아니다). 엔진은 백엔드에 있는 stateless 순수함수로, 정체 streak 같은 상태를 저장하지 않고 매번 DB 히스토리에서 derived로 재계산한다 — 확정 스펙의 derived 원칙과 정확히 일치한다.

**중요한 정정:** 확정 심화설계 문서는 "프론트에 백엔드 엔진과 동형인 TS 쌍둥이 엔진(lib/progression)이 있다"고 전제하지만, **실제로는 그런 엔진이 프론트에 없다**(순수 타입 정의뿐, 계산 로직 없음). 따라서 "Swift로 가면 TS 엔진을 못 쓴다"는 손실은 사실상 없다. 엔진 재사용은 언어 무관하게 이미 서버 REST 하나로 통일돼 있다.

### 4-2. 엔진을 서버에 두는 게 기본값이어야 하는 이유

1. **검증 인프라가 서버에만 있다** — 백엔드에 JUnit(엔진 테스트·세트 로직 테스트)이 있고 h2 프로파일로 골든 케이스 교차검증이 된다. 엔진을 Swift 클라로 옮기면 이 자산을 버리고 XCTest로 다시 짜야 하는데, `simctl` hang 이력상 자동검증이 가장 약한 곳이 바로 여기다.
2. **AI 리포트가 엔진 결과를 "fact"로 주입받아 환각을 막는다** — 엔진이 클라로 가면 AI에 먹일 fact를 다시 서버로 올려야 해 왕복이 생긴다.
3. **규칙 수정 시 앱스토어 심사 없이 서버 배포로 즉시 반영된다** — 근비대 규칙은 계속 튜닝될 영역이라 이 민첩성이 크다.

### 4-3. 확정 엔진은 아직 미구현 — 어차피 신규 개발

현재 서버 SetType엔 TOP_SET/BACKOFF가 없고, ExerciseGoal엔 progressionRule·anchorTier·ruleParams JSON·gateWindow가 없다. 즉 확정 심화설계의 4-Strategy 엔진·이중게이지·랩업 묶음·추세 streak는 전부 신규다. **이 신규 개발은 Swift 전환과 무관하게 어차피 해야 한다.** 그러니 엔진은 서버에서 v2로 새로 짜고 클라는 순수 소비자로 두는 게 가장 손실이 적다.

### 4-4. 백엔드 유지 범위

**그대로 유지(언어 무관, REST로 Swift에 붙음):** 인증(JWT·애플/구글/카카오 OAuth), 세션/세트 CRUD, 종목 마스터, 통계, AI 주간 리포트.
**신규 스키마로 증분 확장:** SetType에 TOP_SET/BACKOFF 추가(하위호환 유지), ExerciseGoal에 progression_rule·anchor_tier·effort_tracking·gate_window·rule_params(JSON) 컬럼(전부 nullable/DEFAULT라 무중단), WorkoutSet에 effort·superset_group·duration_sec, ProgressionEngine을 `evaluate(goal, history) → ProgressionResult(axes·gate·suggestion)` Strategy 4종으로 리팩터(기존 compute는 어댑터로 남겨 AI fact·기존 DTO가 안 깨지게).
**P0로 최우선:** 주간 부위 볼륨 집계 교정 — 현재 워밍업만 제외하고 sets=3 하드코딩이라, TOP_SET/BACKOFF/NORMAL만 하드셋으로 세고 WARMUP/DROP은 제외하며 targetSets를 실제 산출하도록 고친다. 이 한 레이어가 가변머신·종목교체·시간부족 케이스를 자연히 흡수한다.

### 4-5. 데이터 흐름 — 로컬 잠정 + 서버 확정 (하이브리드)

헬스장은 지하·데드존이 흔해 "기록 중 끊김"이 실사용 리스크다. Swift로 새로 짜는 건 로컬-우선을 넣을 기회다. 권장 모델:

- 세트/세션은 로컬(SwiftData)에 **즉시** 쓰고 `updated_at`+dirty 플래그로 백그라운드 sync.
- 게이지·게이트 판정은 2단계 — **로컬 히스토리로 즉석 잠정표시(단순 산술: 총반복/목표 채움) + 서버 확정 판정으로 교체.**
- 경계선 긋기: **단순 산술(게이지 채움)만 클라, 게이트·추세·증량제안·랩업 같은 판정은 서버 확정으로 제한.** 이러면 오프라인 UX와 단일 판정소스를 동시에 지킨다(두 값이 잠깐 다를 수 있으니 "확정 중" 표기 UX만 설계).
- **회복 데이터는 요약값만 opt-in으로 서버 업로드**(원자료 아님) → 디로드/단발할인 신호 근거로만. 게이트는 절대 자동으로 안 바꾼다.

### 4-6. 충돌·멱등성·마이그레이션

- 세트는 클라 생성 UUID(멱등키)로 POST해 재전송 중복을 서버가 무시하게 한다. 세션 필드(제목·메모)는 last-write-wins.
- 증량 확정·trendStreak 롤백은 서버가 단일 소스라 충돌이 없다("증량 되돌리기" 시 streak 롤백도 서버 처리).
- 마이그레이션 부담은 거의 없다 — 타겟이 본인 1명이고, 서버 신규 컬럼이 전부 nullable/DEFAULT라 기존 목표는 DOUBLE_PROGRESSION으로, 기존 세트는 NORMAL로 자동 폴백한다.

## 5. 단계적 이행 로드맵

무엇부터 하느냐. 리스크가 낮은 서버·코어부터, 검증이 어려운 워치·HealthKit을 맨 뒤로 둔다.

- **0단계 — 서버(Swift 착수 전에 병행 가능):** 주간 볼륨 집계 P0 교정 + SetType TOP_SET/BACKOFF + ExerciseGoal 신규 컬럼 + ProgressionEngine을 evaluate→ProgressionResult Strategy로 리팩터(DOUBLE만 먼저 이전, 동작 동일을 JUnit으로 검증). 이건 언어 결정과 무관하게 지금 시작할 수 있다.
- **1단계 — 인증:** Swift에서 애플 로그인 우선, 그다음 카카오/구글. 기존 /auth REST 그대로.
- **2단계 — 운동 중 화면:** 세션/세트 CRUD 붙이고 세트행 직전값·듀얼 게이지(서버 ProgressionResult 소비).
- **3단계 — 엔진 v2 완성:** TOP_SET_BACKOFF·TOTAL·RIR Strategy 서버 추가, 클라는 axes/gate/suggestion만 렌더.
- **4단계 — 데일리 랩업:** 세션 종료 트리거 + 증량 묶음 + 즉시 확정.
- **5단계 — 라이브 액티비티 + HealthKit 읽기(P1):** 휴식타이머 라이브 액티비티, 수면·HRV opt-in 읽기 → 디로드/단발할인 신호 보강(자동 처방 금지).
- **6단계 — 애플워치 앱(P2):** WorkoutKit 세션 + 햅틱 휴식타이머, WatchConnectivity로 폰 동기화. 위젯.

엔진·랩업까지는 서버 확정 기반이라 리스크가 낮고, 워치·HealthKit은 가치는 크지만 검증 난도가 높아 마지막이다.

## 6. 사용자 판단이 필요한 결정점

가장 큰 방향(전환 여부)부터. 각 옵션의 장단점을 그대로 적는다.

### 결정 1 — Expo 유지 vs SwiftUI 전면 전환

이 결정의 진짜 기준은 "Swift가 더 좋은가"가 아니라 **"애플워치·수면을 정말 만들 것인가"**다.

- **Expo 유지(신규 레포도 RN).** 장점: 18.6K줄 자산과 TS 배관(카카오·API·유틸) 재사용, Expo Go 즉시 폰테스트 + Claude 무인 Web 검증 파이프라인 유지, 리모트 Mac 환경과 마찰 최소, 확정 스펙 MVP를 가장 빠르게 완성, 워치/수면은 react-native-health로 후속 보강 가능. 단점: 진짜 네이티브 워치 단독앱·라이브 액티비티·백그라운드 심박은 브리지로 우회해야 해 매끄러움이 떨어짐, 워치를 붙이는 순간 Expo Go는 못 쓰고 dev build가 필요.
- **SwiftUI 전면 전환.** 장점: 애플워치·HealthKit(수면·HRV) 최상급 통합, 라이브 액티비티·다이나믹 아일랜드·위젯 1급 지원, 회복 기반 디로드 같은 차별화 극대화, 백엔드에 애플 로그인 완비라 auth 재구현 거의 불필요, 서드파티 의존성 최소. 단점: 프론트 18.6K줄 전량 재작성 + TS 배관 폐기, Swift 첫 학습곡선, 리모트 Mac+아이폰으로 Xcode 빌드·서명 마찰 큼, Expo Go 즉시 테스트와 Claude 무인 화면검증 둘 다 상실(simctl hang).

**권고:** 전환의 순이득이 사실상 "워치 네이티브 접근" 하나인데 그마저 하이브리드로 대체 가능하고, 리모트 Mac+아이폰 환경에서 Xcode·서명·검증상실 비용이 확정적으로 크다. **기본값은 Expo 유지.** 단 아래 결정 2에서 "워치를 중심 차별화로 삼겠다"가 확정되면 Swift가 정당해진다.

### 결정 2 — 애플워치·수면을 "중심 차별화"로 둘 것인가, "보조 데이터"로 둘 것인가

이게 사실상 결정 1을 좌우하는 상위 질문이다.

- **중심 차별화(운동 자동감지·손목 단독 로깅·회복 중심).** 장점: Strong류가 약한 지점 정면공략, 게임+진지데이터 방향에 강한 무기, 이 경우 SwiftUI 전환이 정당해짐. 단점: MVP가 무거워지고 완성 지연, 워치 UX·페어링·백그라운드 배터리 등 새 난제, 리모트 개발환경에서 워치 테스트가 특히 어려움.
- **보조 데이터(수면/HRV로 디로드 신호만 보강).** 장점: 확정 스펙 MVP를 워치 없이 먼저 완성, Expo 유지한 채 후속 추가 가능, 리스크 격리. 단점: 워치 단독앱·라이브 액티비티 같은 강한 연출은 못 함, 차별화 임팩트가 약함.

**권고:** **보조 데이터로 시작.** 확정 스펙 어디에도 워치·수면이 필수 의존으로 들어가 있지 않다(듀얼게이지·게이트·랩업·증량제안은 전부 세트 입력만으로 돈다). MVP를 먼저 굳히고 실사용에서 수면/HRV가 정말 값을 내는지 검증한 뒤, 중심으로 승격할지 결정하는 게 리스크 대비 낫다.

### 결정 3 — 회복 기능을 어느 시점에 넣을 것인가

- **MVP에 포함.** 장점: 네이티브 전환 명분을 첫날부터 실현, 사용자가 Swift에 끌린 이유를 바로 검증. 단점: 정체·디로드 판정 로직이 아직 안 굴러가면 회복 데이터를 얹을 "자리"가 없어 선후가 뒤바뀜, 베이스라인 2주 필요라 초기엔 어차피 침묵, MVP 범위·검증 비용 급증.
- **P1 후속.** 장점: 코어(확정 8+7 결정)를 먼저 완성해 회복이 얹힐 판정 로직을 갖춘 뒤 추가, MVP 범위 통제, 회복 없이도 앱이 완결적. 단점: 네이티브 전환 명분이 초기엔 약해 보임.

**권고:** **P1 후속.** 회복 데이터는 정체·디로드 판정을 "강화"하는 보조축이라, 그 판정 로직이 실제로 돌아가는 코어가 먼저 있어야 얹을 자리가 생긴다. 단 HealthKit 읽기 배관·권한 온보딩·베이스라인 저장소는 아키텍처 설계 시점에 미리 자리를 비워둔다.

### 결정 4 — 워치 "앱"까지 만들 것인가, HealthKit "읽기"만 할 것인가

- **HealthKit 읽기만.** 장점: RN EAS Build로도 가능해 TS 배관 재사용 이득 유지, 개발·검증 비용 작음, 자동검증 가능 영역, 회복 코칭 가치의 대부분을 이미 확보. 단점: 손목 햅틱 휴식타이머·손목 세트확정 같은 "헬스장 물리적 편의"는 못 줌, 애플 생태계 밀착 정체성은 약함.
- **워치 앱까지.** 장점: 헬스장 3초룰에 최적(폰 안 꺼냄), 햅틱 휴식·활동링 기록으로 몰입 극대화, 애플 전용 앱의 진짜 차별화. 단점: 사실상 두 번째 앱(별도 UI·WatchConnectivity·별도 QA), 범위·유지보수 크게 증가, Claude 자동검증 불가로 회귀 위험, 타겟이 워치 항시착용자가 아님.

**권고:** **HealthKit 읽기만 먼저(P1), 워치 앱은 P2.** 읽기만으로 회복 코칭 가치의 대부분을 얻고 비용·검증 리스크가 작다. 워치 앱은 코어가 안정되고 "손목 편의"가 실제 병목으로 확인된 뒤 투자.

### 결정 5 — 엔진을 어디에 둘 것인가

- **서버 단일.** 장점: JUnit·골든검증·AI fact 자산 그대로, 규칙 튜닝을 앱 심사 없이 서버 배포로 즉시, 워치·폰·AI가 같은 판정 공유. 단점: 오프라인에선 서버 응답 전까지 게이트가 안 뜸(잠정 산술표시로 완화 필요), 매 판정에 네트워크 왕복.
- **클라 잠정 + 서버 확정(하이브리드).** 장점: 단순 산술(게이지 채움)만 클라 즉석, 게이트·추세·증량·랩업은 서버 확정으로 교체 — 오프라인 UX와 단일 판정소스를 동시에. 단점: 두 값이 잠깐 다를 수 있어 "확정 중" 표기 UX 설계 필요.
- **클라 단일(Swift).** 장점: 완전 오프라인·즉각 반응. 단점: JUnit 검증자산 폐기·XCTest 재작성(자동검증 최약 지점), AI fact를 다시 서버로 올려야 함, 규칙 수정마다 앱 심사, 워치와 규칙 이중구현 위험.

**권고:** **클라 잠정 + 서버 확정 하이브리드.** 코어 판정(게이트·추세·증량·랩업)은 서버 단일 소스로 두어 검증·AI·워치 공유 이점을 지키고, 클라는 오프라인에서 총반복 대비 게이지 같은 단순 산술만 잠정표시. 회복 신호는 자동 처방 없이 어드바이저리로만.

### 결정 6 — (Swift로 갈 경우) 검증 상실을 어떻게 감당할 것인가

- **사용자 폰 수동검증에 전면 의존.** 장점: 추가 인프라 불필요. 단점: 리팩터링 안전망 상실, Claude 왕복 느려짐, 아이폰으로 빌드로그 읽기 나쁨, 회귀 놓치기 쉬움.
- **xcodebuild CLI + 스냅샷 테스트 파이프라인 구축.** 장점: 일부 무인검증 회복(빌드성공·유닛·스냅샷), Claude가 CLI로 돌릴 여지. 단점: 구축 비용 큼, SwiftUI Preview·simctl은 여전히 자동화 셸에서 불안(hang 이력), 픽셀검증은 실기기 필요.

**권고:** 이 검증 문제 자체가 Expo 유지의 강한 근거다. Swift로 간다면 최소한 xcodebuild+스냅샷 파이프라인을 **먼저** 세팅하고 진입해야 하며, 그 세팅 비용까지 전환 비용에 포함해 판단할 것.

## 부록 — 한 줄 결론

**지금 시점의 현실적 기본값은 "Expo 유지 + 확정 스펙 MVP 완성 → react-native-health로 수면/HRV 후속 보강".** SwiftUI 전면 전환은 "애플워치를 제품의 중심 차별화(운동 자동감지·손목 단독 로깅·회복 중심)로 삼겠다"는 확신이 설 때, 그리고 xcodebuild+스냅샷 검증 파이프라인을 먼저 세운 뒤에 재평가할 2단계 결정이다. 백엔드(엔진·인증·통계·AI)는 어느 쪽이든 그대로 산다.
