# 점진적 과부하(Progressive Overload) 연구 종합 및 앱 적용안

> 작성 기준: 동료심사 메타분석/RCT(논문) + 저명 코치 실전 견해(전문가) + 검증된 훈련 시스템 해부(시스템)를 교차검증해 조립.
> 목적: GymTracker의 점진적 과부하 룰 엔진을 근거 기반으로 보강하기 위한 단일 참조 문서.
> 작성일: 2026-06-28

---

## 0. 한 문장 결론

근비대 점진적 과부하의 기준 변수는 **'무게'가 아니라 '0~3 RIR로 수행된 주간 유효 볼륨'**이며, 무게 증가는 세트를 계속 힘들게 유지하기 위한 *도구*일 뿐이다. GymTracker는 이미 검증된 골격(double progression + 분류별 메서드 + 3세션 10% deload, 상용 Hevy Trainer·StrongLifts와 일치)을 갖췄고, 보강 포인트는 명확하다.

## 1. 과학적 합의 (세 리서치가 일치한 핵심)

| # | 합의 결론 | 대표 근거 |
|---|---|---|
| 1 | 근비대 1차 동인은 **기계적 장력**. 근손상은 부산물이지 목표 아님 | Schoenfeld 2010 |
| 2 | 근비대를 예측하는 변수는 **무게가 아니라 힘든 세트의 주간 볼륨**. 무게는 근력의 기준 변수 | López 2021 (근비대 SMD 0.12 비유의 / 근력 SMD 0.60), Currier 2023 |
| 3 | 볼륨-근비대는 용량반응 관계. **세트 1개당 +0.37% 성장**, 수확 체감 있음 | Schoenfeld·Krieger 2017, Pelland 2026 |
| 4 | Rep 범위는 **5~30회 넓게 동등**(40~85% 1RM). ~20% 1RM 초저부하만 열등 | Morton 2016, Lasevicius 2018 |
| 5 | 실패 근접도(RIR)는 **품질 게이트**: 0~3 RIR면 충분, 완전 실패 불필요 | Refalo 2023, Robinson/Pelland 2024 |
| 6 | 주당 빈도는 **주간 볼륨을 맞추면 독립 동인 아님**. 빈도는 볼륨 분배 수단 | Schoenfeld 2019 |
| 7 | RIR/RPE 자기조절은 %1RM 대비 **최소 동등**. 단 실패에서 멀수록·초보일수록 부정확 | Zourdos 2016/2021, Helms 2018 |
| 8 | **디로드는 근비대를 촉진하지 않는다** (오히려 약한 역효과). 회복·지속성 용도 | Coleman 2024 (디로드군 근비대 무이득, 근력은 연속훈련 우위) |

### 레벨별 progression (전문가 합의)
- **초급**: 가능한 한 매 세션 무게 증량(선형). RIR은 받되 **증량 결정엔 거의 반영 말 것**(초보는 실패까지 4~5렙 과소평가 — Steele 2017). 기술/풀 ROM 우선. (Rippetoe: "초보=지난 운동 기준")
- **중급**: 주 단위. Double progression + 소량 증량. RIR 신뢰도 상승 → 증량 트리거 반영 시작. (Rippetoe: "중급=지난 한 주 기준")
- **고급**: 월~블록 단위. 세션 증량 기대 낮추고 볼륨(MEV→MRV 세트 추가)·RIR 자동조절 중심. 정체 판정 창을 길게. 디로드 정기화. (Rippetoe: "고급=지난 한 달 기준")

## 2. 현재 앱 룰 엔진 진단

**코드 위치**: `lib/progression/progressionEngine.ts`, `progressionRules.ts`, `progressionTypes.ts`, `lib/overload.ts` (+ 백엔드 `gymtracker-backend/.../overload/ProgressionEngine.java`)

### 잘 하고 있는 것 (검증 시스템과 부합)
- **상단 rep range를 모든 세트에서 채우면 증량** = 정통 double progression. Hevy Trainer 상용 구현과 일치.
- **effort(NEAR_LIMIT/FAILURE면 증량 보류)** = RPE/RIR autoregulation의 단순형. RPE 7~9 권고와 부합.
- **3세션 연속 정체 → STALL_REVIEW → 5~10% 감량** = StrongLifts 기준과 정확히 일치.
- **종목 분류별 rep range**(머신 8~12, 고립 12~20)는 코칭 컨센서스와 대체로 부합.
- **무게 고정 + 총 반복수 +1~2 처방**은 근비대 관점(볼륨이 기준)에서 올바른 방향.

### 보강이 필요한 것 (근거와 어긋남)
1. **[코드 버그성] 상/하체 증량폭 차등이 본선에서 무력화.** `progressionRules.ts`엔 하체 5kg/상체 2.5kg가 있는데 `progressionEngine.ts:115`가 **분류 무관 항상 2.5kg 하드코딩**. 모든 검증 시스템(SS·5/3/1·nSuns)은 하체를 상체의 ~2배로 증량.
2. **[죽은 분기] userGoal(strength/hypertrophy)이 무효.** 둘 다 6~10렙으로 귀결. strength면 바벨 3~6렙이어야 하는데 분기가 죽음.
3. **[개념 부재] mesocycle 볼륨 점증 없음.** 무게/렙만 다루고 주간 세트 점증(RP MEV→MRV)이 없다. 고립/머신은 무게보다 볼륨 증가가 자연스러운데 출구가 없음.
4. **[개념 부재] 마이크로로딩 없음.** 고립/머신에 5kg 핀 점프는 부하 대비 충격이 큼(예 25kg에 20%). 0.5~1.25kg 옵션 부재 → 고립 증량이 사실상 막힘.
5. **[죽은 스테이지] INCREASE_LOAD / CONSOLIDATE가 타입만 있고 전이 안 됨.** 증량 직후 적응 단계가 비어 곧바로 BUILD_REPS 복귀.
6. **[지표 오정렬] Epley 추정 1RM을 근비대 진행 지표로 사용.** 1RM은 근력 지표 — 근비대 진행을 잘 대표 못 함. 1RM 정체해도 볼륨↑면 근비대 진행 중일 수 있음. 게다가 Epley는 고반복(>10~12)에서 1RM 체계적 과대추정.
7. **[판정 창] 정체/디로드 판정이 세션 단위.** 진짜 정체는 다주~다월 객관적 정체(Nuckols·Israetel). 세션 단위면 정상 변동을 정체로 오판.
8. **[포지셔닝 오류] DELOAD를 '성장 재민감화'로 가정.** Coleman 2024은 근거 부족. 회복/지속성 용도로 재포지셔닝 필요.

## 4. 앱 적용 액션 (우선순위)

### P0 — 저비용·고임팩트 (이미 코드/데이터 있음, 연결만)
1. **`progressionEngine.ts:115` 고정 2.5kg → 분류·상하체 차등 테이블로 교체** (위 3장 표). 가장 명확한 버그성 수정.
2. **증량 트리거 기본값을 '2세션 연속/RPE 여유'로** 변경 ('한 세션 성공'은 옵션 유지). 근거: 자기조절.
3. **죽은 분기 복구**: strength rep range(바벨 3~6), INCREASE_LOAD 스테이지(증량 직후 1세션 하단 재확립).

### P1 — 중간 노력, 핵심 차별화
4. **고립/머신 마이크로로딩 + 세트 점증(mesocycle 볼륨)** 도입. 무게 증량 여지 적은 분류의 진짜 진행 동력.
5. **2단계 정체 처리(GZCL식)**: 1차 정체 = 세트 추가/rep 재조정 → 그래도 막히면(재차 3세션) 10% deload. 한 번 막혔다고 바로 감량 안 함 → UX 부드러움.
6. **레벨별 RIR 반영도 차등**: 초보는 RIR을 '학습용'으로 표시하고 증량 결정 반영도 낮춤(Helms 명시). 중급+부터 트리거 반영.

### P2 — 지표·포지셔닝 개선
7. **주간 유효 볼륨을 1차 진행 지표로 노출**(1RM은 보조로 강등). 부위별 직접/간접 세트 구분.
8. **디로드를 '성장' 아닌 '회복/지속성'으로 재포지셔닝.** 자동 강제보다 피로 신호/사용자 선택 기반.
9. **정체 판정을 세션 단위 → 다주 누적(2~4주 정체 + 같은 무게서 RPE 상승)으로** 게이팅.

### P3 — 신규 기능 (흔한 실수 방지)
10. DELOAD_OR_RESET에 **원인 진단 분기**(기술/약점/부상/프로그래밍 — Nuckols 4분류) + 변수 교체 제안.
11. 증량 시 **반복수 급감 경고**(폼 붕괴 방지), MRV 근처 회복 지표 기반 디로드 알림.

## 5. 불확실 / 논쟁 지점 (과신 금지)

- **볼륨 상한**: Pelland 2026(상한 없는 체감) vs Enes 2024(30+세트 정체). 개인·근육·회복 의존. 고볼륨은 낮은 확실성.
- **Volume landmarks(MV/MEV/MAV/MRV)**: Israetel/RP의 코칭 휴리스틱이지 직접 검증 모델 아님. RP 본인도 "출발점"이라 명시. 앱에선 '실용 가이드'로 라벨링.
- **Double progression**: 원리는 확실하나 이 특정 알고리즘 vs 타 진행법 직접 RCT 없음(관행).
- **'Effective reps'(마지막 ~5렙만 자극)** 엄격 모델은 미지지. 실패 근접도는 비선형 문턱 효과이지 반복 카운팅 아님.
- **빈도-근력**: 근비대엔 무관 합의, 근력엔 소폭 독립 효과(강도 약함).

## 6. 차별화 인사이트

Strong/Hevy(소비자 앱)는 대부분 **"지난 기록 표시"에 그치고 다음 무게를 자동 처방하지 않는다.** 자동 처방은 Stronglifts·Dr. Muscle·Hevy Trainer 같은 코칭 계열만 한다. 따라서 GymTracker의 **자동 처방 + (신규) 원인 진단 분기**는 실제 시장 공백이다. 단, 처방의 신뢰를 위해 위 P0~P1(차등 증량, 마이크로로딩, 2단계 정체)이 선행되어야 한다.

## 출처 신뢰도 메모

- 논문(Schoenfeld 2017, Coleman 2024 등) 핵심 수치는 원문 직접 재확인 완료.
- 전문가: RP·Nippard·Helms/Steele는 1차 출처 직접 확인. Stronger by Science·Legion 일부는 403 차단으로 검색 스니펫 기반(중상 신뢰).
- 시스템: nSuns 상/하체 구분은 T1 표가 아닌 T2 보조운동 적용임을 검증. RP 등근육 세트 수치는 출처 충돌로 신뢰도 낮음(가슴·대퇴 표는 일관).

# 7. 4인 패널 토론 최종 결론 (실행계획)

> 효용·코드·리서치 3인 패널 + 사회자가 2라운드 토론. 코드를 직접 추적해 1장의 적용안 일부를 **정정**했다. 아래가 확정 실행계획이다.

## 토론으로 뒤집힌 사실 (중요)
- **1장의 P0-1 "상/하체 차등 증량 버그"는 허수였다.** 그게 가리킨 `lib/progression/progressionEngine.ts`는 **죽은 코드**(테스트에서만 import). 진짜 엔진은 백엔드 `gymtracker-backend/.../overload/OverloadService.java` + `ProgressionEngine.java`이며, 상/하체 차등은 `OverloadService.java:80`에서 **이미 동작 중**(incUpper/incLower).
- **진짜 버그 발견**: `OverloadService.java:590`에서 `ProgressionEngine.compute(..., true)`로 **`readyByEffort`가 상수 `true`로 박제**돼 있다. 그 결과 사용자가 온보딩에서 고른 증량 트리거(single/2세션/rpe)도, RIR/effort도 **엔진에 전혀 반영 안 됨**. 항상 'single 폴백'으로만 처방. 사용자 선택이 무시되는 신뢰 배신 버그.
- **볼륨/1RM 데이터는 이미 계산됨**(`db/api/stats.ts` getVolumeStats/get1RMHistory, getRecords). 화면 노출만 안 됐다 → "지표 교체"는 비싼 작업이 아니라 "노출 추가"라 S급.

## 명시적 보류 — 이번 사이클 "안 한다" (만장일치)
- **자동 디로드 강제 / 디로드 '성장 재민감화' 재포지셔닝** — Coleman 2024상 행동 변화 0, 효과크기 작음. 단 디로드 UI에 "성장" 인과 카피를 *새로 쓰지는 말 것*.
- **mesocycle 풀구현 / MEV·MRV 정밀 수치화** — 검증 모델 아닌 휴리스틱. 세트추가 출구(P0-3)까지만.
- **1RM 강등(볼륨으로 교체)** — 병기까지만. 화폐는 1RM 유지.
- **RIR 강제 입력** — 선택 입력 + 보조 반영만. 세트마다 강제 금지.
- **다주 캘린더 정체 판정 / 레벨별(초·중·고) 분기** — 현 "같은 무게 3회" 게이팅으로 충분. 타겟이 중급 1명이라 분기 불필요.

## 핵심 합의 한 줄
이 앱의 효용은 "정확하고 신뢰할 수 있는 다음 세트 처방"이다. 화려한 과학(mesocycle·MRV·자동디로드)이 아니라 **(1) 사용자가 고른 트리거가 실제로 작동하게 하고(P0-2), (2) 고립/머신 증량 막힘을 뚫고(P0-3), (3) 처방 기준값 버그를 없애는 것(P0-1)** 에 집중한다.
