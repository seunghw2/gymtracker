# GymTracker 비판 검토 (CRITIC_REVIEW)

> 작성: Critic Agent. 검토 대상 5개 문서(APP_AUDIT, PROGRESSION_PROGRAMS, USER_INPUT_SPEC, PROGRESSION_ENGINE_SPEC, UX_SCREEN_SPEC)를 모두 읽고 비판.
> 기준: 타겟은 **중급+ 본인 1명**. 입력 최소화·헬스장 3초 이해·기준 부족 시 분석 금지·비현실적 목표 금지가 철칙.
> 관점: 과잉설계를 날카롭게 쳐내고, 5개 문서가 서로/철칙과 충돌하는 지점을 잡아낸다. 코드 미수정.
> 작성일: 2026-06-29

---

## 0. 한 줄 진단

5개 문서는 "현 엔진을 정직하게 재명세"하는 데까진 좋다. **그러나 전부 합치면 1인 중급 사용자에게 과한 기능 목록이 된다.** 9 stage 머신 완성·2단계 정체 안전망·임계 후 가중 전환·컨디션 입력·리포트 AI 브리핑·종목 상세 진단 카드 등은 "있으면 좋은 것"이지 "Strong을 더 편하게 대체"하는 MVP가 아니다. 가장 큰 모순은 **RIR을 살리자는 P0**와 **입력 최소화 철칙**이 정면 충돌하는데 5개 문서 중 누구도 이걸 끝까지 따지지 않은 것이다.

## 1. 사용자에게 너무 귀찮은 입력

### 1-1. 매 운동 "마지막 세트 노력도" 버튼 — 사실상 매 종목 입력
- **출처**: USER_INPUT_SPEC §5(노력도 버튼 4개), APP_AUDIT §4·§15(RIR P0로 "살려라"), PROGRESSION_ENGINE_SPEC §6-6.
- **비판**: 문서들은 "마지막 작업 세트 1회만"이라 가볍다고 주장하지만, 한 세션에 종목이 5~6개면 **종목마다 1번 = 세션당 5~6번 팝업**이다. 게다가 RIR 추정은 중급도 들쭉날쭉하고(PROGRESSION_PROGRAMS §6이 직접 인정: Steele 2017), 입력해도 "여유 많음/2~3개/거의 한계/실패" 4단계의 노이즈가 증량 판정을 흔든다. 매 세트 완료 후 바텀시트가 뜨는 건 운동 흐름을 끊는다.
- **모순 지적**: USER_INPUT_SPEC은 같은 문서 안에서 "입력 최소화가 1원칙"이라 해놓고 P0로 RIR 입력을 **추가**하라고 한다. APP_AUDIT조차 §13-1에서 "안 그러면 차라리 안 묻는 게 정직"이라 적었다 — 이게 정답에 가깝다.
- **수정 제안**: MVP에서는 노력도 팝업을 **제거**한다. 증량 판정은 이미 가진 데이터(반복수×무게×세트)와 two_sessions 트리거로 충분하다(PROGRESSION_PROGRAMS 결론도 "Double Progression 입력은 weight·reps뿐"이라 함). RIR은 모델·전송·엔진 3단을 다 만들어 "진짜 판정에 쓸" 확신이 설 때 후속으로. 그 전까진 받지도 말 것.
- **심각도**: 높음 / **MVP 반영**: 제외(노력도 입력 자체를 MVP에서 뺀다)

### 1-2. 운동 종료 후 "컨디션" 선택형 입력
- **출처**: USER_INPUT_SPEC §6(컨디션 좋음/보통/피곤 선택형 1개 옵션).
- **비판**: 문서 스스로 "현재 없음, 추가 시에도 필수 아님"이라 했는데 **그러면 왜 넣나**. 엔진이 컨디션을 소비하는 경로가 어디에도 없다(APP_AUDIT 어느 §에도 컨디션 입력→판정 경로 없음). 소비처 없는 입력은 RIR과 똑같이 "받고 버리는" 죽은 입력이 된다.
- **수정 제안**: MVP에서 완전 삭제. 종료는 읽기 전용 요약만.
- **심각도**: 중간 / **MVP 반영**: 제외

### 1-3. 온보딩 "증량 트리거" 3지선다
- **출처**: USER_INPUT_SPEC §1(증량 트리거 single/two_sessions/rpe), GoalSetting.
- **비판**: 중급 1명이 온보딩에서 "single vs two_sessions vs rpe"를 의미 있게 고를 수 있나? rpe는 데이터 없어 single 폴백이고(여러 문서 인정), single과 two_sessions 차이는 사용자가 체감하기 전엔 모른다. 4개 온보딩 입력 중 가장 "이해 비용 높고 효용 불명확"한 항목.
- **수정 제안**: MVP 온보딩에서 트리거 선택지를 **숨기고 two_sessions 고정**. 설정 고급 옵션으로만 노출. 온보딩 입력 4→3개.
- **심각도**: 중간 / **MVP 반영**: 기본값 고정(선택 UI 제외)

## 2. 점진적 과부하 과적용

### 2-1. 9 stage 머신 완성 요구
- **출처**: PROGRESSION_ENGINE_SPEC §4·§6(INCREASE_LOAD/CONSOLIDATE/HOLD_OR_REPEAT/DELOAD_OR_RESET 추가), APP_AUDIT §15 P1.
- **비판**: 현 5 stage로 중급 1명에게 필요한 흐름(기준생성→반복쌓기→증량준비→정체점검)은 다 커버된다. CONSOLIDATE/INCREASE_LOAD는 "증량 직후 1~2세션 적응"인데, 사용자 입장에선 그냥 BUILD_REPS로 보여도 행동이 동일하다("이 무게로 한 번 더"). 4개 stage를 추가하면 코드·테스트·카피가 늘고 stageLabel 종류가 많아져 오히려 "지금 무슨 단계지?" 혼란이 커진다.
- **수정 제안**: MVP는 5 stage 유지. CONSOLIDATE/INCREASE_LOAD는 **stage를 새로 만들지 말고** READY_TO_INCREASE/BUILD_REPS의 카피로 흡수("증량했으면 이 무게로 한 번 더 안정화"). DELOAD만 정체 안전망으로 후속 검토.
- **심각도**: 높음 / **MVP 반영**: 제외(5 stage 유지, 신규 4 stage 안 만듦)

### 2-2. 임계 후 가중 전환(맨몸)
- **출처**: PROGRESSION_PROGRAMS §184·표, PROGRESSION_ENGINE_SPEC §3.4·§6-7.
- **비판**: "풀업 8~10 도달 후 +2.5kg 가중으로 Double Progression 전환"은 가중벨트 장비·임계 판정·무게 입력 전환 등 특수 케이스 분기다. 종목 몇 개나 해당되나? 맨몸 진행(총 반복수 +1)으로 충분하고, 가중이 필요해지면 사용자가 그냥 무게를 치면 된다(이미 무게0 허용).
- **수정 제안**: 엔진 분기 추가 불필요. MVP 제외.
- **심각도**: 낮음 / **MVP 반영**: 제외

## 3. 비현실적 오늘 목표

### 3-1. 머신 bump=[2,3] — "총 반복수 +2~3" 과함
- **출처**: PROGRESSION_ENGINE_SPEC §2.6·§3.2(`buildBump` 머신 [2,3]), PROGRESSION_PROGRAMS §4.
- **비판**: 철칙(§0-4)은 "오늘 목표 = 직전 총 + 작은 bump(무게+1~2)"인데, 머신만 [2,3]으로 더 크게 잡는다. 직전 3세트 합이 30회였다면 오늘 32~33회를 요구하는 셈. 머신이라고 회복이 빠른 근거는 약하고, "작은 bump" 원칙과 자기모순이다. 큰 점프 금지하자면서 머신은 큰 점프를 허용한다.
- **수정 제안**: 머신도 +1~2로 통일하거나, 최소한 "전 세트 동시 +1"이 아니라 "세션 총합 +1"로 더 보수적으로. 분류별 차등 bump는 효용 대비 복잡도가 높다.
- **심각도**: 중간 / **MVP 반영**: 반영(머신 bump를 +1~2로 통일)

### 3-2. 단기 목표 `max(8, ceil(직전총×1.5))`
- **출처**: PROGRESSION_ENGINE_SPEC §3.4(BODYWEIGHT shortTerm).
- **비판**: 직전 총 3회면 단기 8회(×2.7배), 직전 6회면 단기 9회. 곱하기 1.5는 근거가 약하고 케이스마다 들쭉날쭉(작은 수에선 +8, 큰 수에선 +3). 카드에 안 띄우고 시트에만 둔다 해도 "단기 8회"가 직전 3회 옆에 뜨면 비현실적으로 보인다.
- **수정 제안**: 단기 목표는 "현 범위 상단×세트수"처럼 **규칙이 명확한 값**으로. 임의 ×1.5 제거.
- **심각도**: 중간 / **MVP 반영**: 반영(단기 산식 단순화)

## 4. 너무 빠른 정체 판단

### 4-1. 정체 기준 이중화(엔진 stalling vs stats plateauWeeks)
- **출처**: APP_AUDIT §14-4·현구현표, UX_SCREEN_SPEC §6(정체 정의 일원화 요청), PROGRESSION_ENGINE_SPEC §4.
- **비판**: 같은 종목에 "엔진=3세션 비증가로 STALL", "리포트=N주 PR없음으로 plateau" 두 정의가 동시에 돈다. 한쪽은 정체라는데 한쪽은 아니면 사용자는 어느 메시지를 믿나. 5개 문서가 모두 "통일해야 한다"고 적었지만 **누구도 어느 정의로 통일할지 정하지 않았다** — 미해결 상태로 남았다.
- **수정 제안**: MVP는 **엔진 stalling 단일 정의로 통일**하고, 리포트의 plateauWeeks 기반 정체 메시지는 **끈다**(리포트는 정체를 말하지 말고 종목 상세 링크만). 정체는 한 곳(엔진)에서만 선언.
- **심각도**: 높음 / **MVP 반영**: 반영(엔진 단일 정의, 리포트 정체 문구 제거)

### 4-2. 세션 단위 3회 비증가 = 정체
- **출처**: PROGRESSION_ENGINE_SPEC §4(STALL_REVIEW: 같은 무게 3회 연속 비증가), PROGRESSION_PROGRAMS §9(단점 인정: 정상 변동 오판).
- **비판**: 세션 단위 3회는 짧을 수 있다. 컨디션 나쁜 날 1번 끼면 3세션이 금방 채워진다. 문서도 "세션 단위 정체 판정은 정상 변동을 오판"이라 인정. setCountSimilar 게이팅이 있어도 변동 노이즈는 남는다.
- **평가**: 다만 STALL_REVIEW의 처방이 "감량"이 아니라 "한 번 더 기록/세트 추가"라서 오판의 피해는 작다. 즉시 디로드만 안 하면 허용 가능.
- **수정 제안**: 3회 유지하되, STALL_REVIEW 첫 진입은 **반드시 "한 번 더 같은 조건 기록"(비교 신뢰 확보)부터**. 감량 단정 절대 금지(2단계 안전망의 1단계만 MVP).
- **심각도**: 중간 / **MVP 반영**: 반영(처방을 재기록 우선으로 고정)

## 5. 기준 기록 부족한데 분석하는 부분

### 5-1. 홈/리포트의 "이번 주 개선" 도트바
- **출처**: APP_AUDIT §2(개선 도트바 imp.done/total), USER_INPUT_SPEC §7(improvements), UX_SCREEN_SPEC §1.
- **비판**: 기록 1~2개뿐인 종목까지 "개선/미개선"으로 집계하면 BASELINE 단계 종목을 개선 모수에 넣게 된다. 철칙(기준 부족=분석 금지)과 충돌. "개선 0/3" 같은 표시가 기준 생성 중인데도 "못했다"처럼 읽힌다.
- **평가**: 문서들이 evaluate에서 BASELINE_CREATED/COMPARISON_DEFERRED를 분리하긴 한다. 다만 **주간 집계 분모(total)에서 baseline·deferred 종목을 빼는지**가 어느 문서에도 명시되지 않았다.
- **수정 제안**: 개선 도트바 분모에서 baseline/deferred 종목 제외 명시. 데이터 없으면 도트바 자체를 "다음 운동부터 추적"으로(이미 일부 구현). 기준 부족 종목은 절대 "미개선"으로 안 셈.
- **심각도**: 중간 / **MVP 반영**: 반영(집계 모수에서 기준부족 제외 명시)

### 5-2. 리포트 3주 미만 신뢰도 낮음 — 그래도 분석은 함
- **출처**: APP_AUDIT §6·UX_SCREEN_SPEC §6(3주 미만 confidence low).
- **비판**: "신뢰도 낮음 배지"를 달면서도 분석 텍스트는 생성한다. 기준 부족이면 분석 대신 기준 생성으로 가야 하는데, 리포트는 "약한 분석 + 배지"로 타협한다. 1인 사용자 초기 몇 주엔 리포트가 거의 공허하다.
- **수정 제안**: 3주 미만이면 리포트 본문 대신 "아직 기록 쌓는 중 — N주 후 첫 리포트" 단일 화면. 약한 분석을 띄우지 말 것. (→ 리포트 MVP 제외와도 연결, §10 참조)
- **심각도**: 중간 / **MVP 반영**: 반영(또는 리포트 자체 MVP 제외)

## 6. 모든 운동을 증량 대상으로 보는 문제

### 6-1. 홈 정렬이 READY_TO_INCREASE를 최상단으로 — "증량 우선" 편향
- **출처**: UX_SCREEN_SPEC §1(증량 준비 종목을 최상단 액션 섹션 승격), PROGRESSION_ENGINE_SPEC §2.11(stageOrder: READY 0순위).
- **비판**: 화면 구조가 "증량할 종목"을 가장 먼저·초록으로 강조한다. 이건 근비대 목표(유효 볼륨이 1차 동인)와 어긋난다. 고립·머신은 증량보다 반복/볼륨 누적이 맞는데, 홈이 "오늘 올릴 종목"을 영웅처럼 띄우면 사용자가 무게 올리기에 과몰입한다. 증량은 결과지 목표가 아니다.
- **수정 제안**: 홈 최상단 섹션 라벨을 "오늘 올릴 종목"이 아니라 "오늘 행동이 필요한 종목"으로 중립화. READY와 STALL을 같은 "확인 필요" 묶음으로. 증량을 시각적 최우선 보상으로 만들지 말 것.
- **심각도**: 중간 / **MVP 반영**: 반영(라벨 중립화, 증량 단독 강조 완화)

### 6-2. 무게0 고립의 "유지" 출구는 좋음 — 다만 log_only 활용 권장 명시 부족
- **출처**: PROGRESSION_ENGINE_SPEC §3.3(무게0 고립 "유지"), USER_INPUT_SPEC §3(log_only).
- **비판**(약): 증량이 의미 없는 종목(스트레칭·재활·코어 일부)을 사용자가 core로 두면 엔진이 계속 "오늘 목표"를 만든다. 자동으로 log_only를 권하는 경로가 없다.
- **수정 제안**: 무게0·증량 비대상 종목은 기본 role을 support/log_only로 자동 배정 강화. 별도 기능 추가는 불필요.
- **심각도**: 낮음 / **MVP 반영**: 반영(자동 role 배정 보수화)

## 7. 피지크/근비대 목표와 안 맞는 종목 우선순위

### 7-1. 1RM/PR을 헤더·홈에 크게 — 근비대엔 부차 지표
- **출처**: APP_AUDIT §5(종목 상세 큰 1RM 헤더), UX_SCREEN_SPEC §3(전체 PR 노출), PROGRESSION_ENGINE_SPEC §2.3(1RM=동기부여 화폐).
- **비판**: 종목 상세가 "큰 1RM 헤더"로 시작한다. 근비대 사용자에게 1RM은 동기부여 화폐일 뿐 1차 지표가 아닌데(문서도 인정), 화면에서 가장 큰 자리를 차지한다. "유효 볼륨/주간 세트"가 더 큰 자리여야 목표와 맞는다.
- **수정 제안**: 종목 상세 헤더를 1RM 단독에서 "최근 작업 무게×반복 + 주간 볼륨" 중심으로. 1RM은 한 칸으로 축소. (이건 표시 우선순위라 코드 변경 작음.)
- **심각도**: 중간 / **MVP 반영**: 반영(1RM 시각 우선순위 낮춤)

### 7-2. 종목별 단계 머신 우선 — 부위 볼륨 균형이 뒤로
- **출처**: PROGRESSION_PROGRAMS §7(Volume Progression 부분 포함), UX_SCREEN_SPEC §1(부위 세트는 종목 카드 아래).
- **비판**: 근비대의 1차 동인은 부위별 주간 볼륨인데, 문서들은 종목별 stage·오늘목표를 주연으로, 부위 볼륨을 조연으로 둔다. "가슴 8/12세트"가 "Lat Pulldown 증량 준비"보다 위에 와야 피지크 목표에 맞다.
- **수정 제안**: 홈에서 "목표 부위 주간 세트 부족분"을 증량 종목보다 **위 또는 동급**으로. 행동 안내를 "부족 부위 채우기" 중심으로.
- **심각도**: 중간 / **MVP 반영**: 반영(부위 볼륨 우선순위 상향)

## 11. AI 과개입

### 11-1. 두 LLM 시스템(리포트 브리핑 + 종목 코치 + 채팅)
- **출처**: APP_AUDIT §5·§6(backend `com.gymtracker.ai`, getExerciseReport, chat).

- **수정 제안**: MVP에서 LLM 경로 전면 제외. 모든 코칭 문구는 엔진의 결정론적 카피(todayTarget/successCondition/nextStep)로만. "왜"가 필요하면 엔진이 근거 문자열을 이미 준다(comparisonReason 등).
- **심각도**: 높음 / **MVP 반영**: 제외(LLM 전면 후속화)

### 11-2. AI가 "다음 주 목표" 생성(`buildNextWeekGoals`)
- **출처**: APP_AUDIT §12(lib/overload buildNextWeekGoals — chat에서만 사용).
- **비판**: 다음 주 목표는 엔진이 stage·nextTarget으로 이미 결정론적으로 산출 가능하다. AI로 다시 만들 이유 없음. chat 제거(§10-2)와 함께 사라짐.
- **수정 제안**: 제거. 다음 목표는 엔진 값 직접 표시.
- **심각도**: 중간 / **MVP 반영**: 제외

## 12. 문서 간 모순/공백 (추가 발견)

1. **RIR 처리 비일관**: APP_AUDIT/USER_INPUT_SPEC은 "P0로 살려라", PROGRESSION_PROGRAMS는 "강제 입력 금지·선택 보조만", ENGINE_SPEC은 "single 폴백 유지·후속". 세 문서가 RIR을 각자 다르게 다룬다 → **MVP는 "받지 않음"으로 통일 필요**(이 리뷰의 결론).
2. **정체 정의 미결**: 5개 문서 모두 "통일하자"만 하고 정의를 안 정함(§4-1). → 엔진 단일로 확정.
3. **bump 자기모순**: ENGINE_SPEC이 "작은 bump" 철칙과 머신 [2,3]를 동시에 둠(§3-1).
4. **홈 대원칙 위반**: UX_SCREEN_SPEC이 "홈 분석 과다 금지" 원칙과 9섹션 홈을 동시에 둠(§8-1).
5. **캐시 필드 이중 소스**: APP_AUDIT §10·§14-3의 ExerciseGoal status/nextTarget 저장값 vs 엔진 재계산값 불일치 — 어느 문서도 MVP 정리 방안을 안 정함. → MVP는 "엔진 재계산이 단일 소스, 저장 캐시 무시" 명시 권장.

## 결론

### MVP에서 반드시 빼야 할 것
1. **노력도(RIR) 입력·팝업** — 받고 버리는 죽은 입력, 운동 흐름 차단, 입력 최소화 철칙 위반(§1-1·§9-1).
2. **리포트 탭(주간 AI 브리핑)** — overload 엔진과 별개·정체 정의 충돌·초기 공허·유지비용 큼(§10-1).
3. **Chat 탭·AI 코치 대화·"대화로 풀기"·buildNextWeekGoals** — LLM 과개입, 1인 중급 잉여(§10-2·§11).
4. **종목 상세 진단 카드·AI 코치 리포트·체크리스트 메모·5탭 차트** — 차트 1개+기록표+시트로 충분(§10-3).
5. **9 stage 완성(INCREASE_LOAD/CONSOLIDATE/HOLD/DELOAD)·임계 후 가중·2단계 디로드** — 5 stage로 충분, 카피로 흡수(§2-1·§2-2).
6. **운동 종료 컨디션 입력·온보딩 증량트리거 선택 UI·추천 루틴 자동생성·주간 코멘트** — 소비처 없거나 효용 불명확(§1-2·§1-3·§8-2·§10-4).

### MVP에서 반드시 넣어야 할 것
1. **세트 입력(무게·횟수·완료) + 직전값 프리필** — 유일한 핵심 반복 입력, 3초 룰의 본체(USER_INPUT_SPEC §4).
2. **운동 중 1줄 목표 배너**("오늘 80kg 총 24회 / 남은 8회")와 **증량 준비 시 다음 무게 자동 프리필(1탭 덮어쓰기 보장)** — 헬스장 3초 룰·입력 절감의 최대 미수확(§9-2·§9-3).
3. **5 stage 결정론 엔진**(NEED_BASELINE/BASELINE_CHECK/BUILD_REPS/READY_TO_INCREASE/STALL_REVIEW) + 직전 비교 기록 기준 오늘 목표(작은 bump +1~2 통일) — 이미 구현, 그대로(§2-1·§3-1).
4. **압축 홈 3블록**(오늘 행동 필요 종목 / 이번 주 출석·부족 부위 / 종목 리스트), 부위 볼륨을 증량 종목과 동급 이상으로(§7-2·§8-1).
5. **기준 부족=분석 금지·비교 보류는 하락으로 표기 금지** — 집계 모수에서 baseline/deferred 제외 명시(§5-1, ENGINE_SPEC 철칙).
6. **정체 단일 정의(엔진 stalling)·첫 정체는 재기록 우선·즉시 감량 금지** — 리포트 정체 문구 제거(§4-1·§4-2).
7. **온보딩 3개(목적·주간횟수·상하체 증량폭, 트리거는 two_sessions 고정)·종료 입력 0** — 입력 최소화(§1-3).

### 핵심 8줄
1. 5개 문서는 현 엔진 재명세는 정직하나, 전부 합치면 중급 1명에게 과한 기능 목록이 되어 과잉설계를 쳐내야 한다.
2. 가장 큰 모순은 RIR(노력도) — APP_AUDIT/USER_INPUT은 "P0로 살려라", PROGRESSION_PROGRAMS는 "강제 금지"라 충돌하는데, MVP는 받고 버리는 죽은 입력이자 흐름 차단이므로 **아예 안 받는다**가 정답이다.
3. 9 stage 완성·임계 후 가중·2단계 디로드는 5 stage 카피로 흡수 가능하니 MVP 제외, stage 종류만 늘려 혼란 키우지 말 것.
4. 정체 정의가 엔진 stalling vs 리포트 plateauWeeks로 이중화됐는데 어느 문서도 통일안을 안 정함 — 엔진 단일로 확정하고 리포트 정체 문구를 끈다.
5. 머신 bump=[2,3]는 "작은 bump" 철칙과 자기모순이라 +1~2로 통일하고, 맨몸 단기목표 ×1.5 산식도 단순화한다.
